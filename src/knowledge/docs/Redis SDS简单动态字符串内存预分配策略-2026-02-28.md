## **Redis SDS 内存预分配策略技术文档**

### **1. 文档概述**

#### 1.1 目的
本文档旨在详细阐述 Redis 中 Simple Dynamic String（SDS，简单动态字符串）数据结构所采用的内存预分配策略。该策略是 SDS 实现高性能字符串操作的核心机制之一。通过深入分析其设计原理、行为规则及优缺点，帮助开发者理解 Redis 在字符串处理上的效率考量。

#### 1.2 目标读者
- Redis 内核开发者
- 希望深入理解 Redis 底层数据结构的工程师
- 对高性能存储系统设计感兴趣的技术人员

#### 1.3 背景
在 C 语言中，传统的以空字符 `\0` 结尾的字符数组（C 风格字符串）在频繁修改（如追加、截断）时存在严重的性能问题：
1.  **内存重分配**：每次长度变化都需重新分配内存，这是一个昂贵的系统调用。
2.  **缓冲区溢出风险**：需手动管理内存，易出错。

为解决这些问题，Redis 自研了 SDS 结构，其内存预分配策略显著减少了内存重分配的次数，从而提升了性能。

### **2. SDS 核心结构**

Redis 定义了多种 SDS 结构（`sdshdr5`， `sdshdr8`， `sdshdr16`， `sdshdr32`， `sdshdr64`），以优化不同长度字符串的内存使用。其通用设计如下（以 `sdshdr8` 为例）：

```c
struct __attribute__ ((__packed__)) sdshdr8 {
    uint8_t len;    // 已使用的字节数
    uint8_t alloc;  // 分配的总容量（不包括头部和结尾的 \0）
    unsigned char flags; // 低3位表示类型，高5位在sdshdr5中用于存储长度
    char buf[];     // 柔性数组，存放实际字符串和结尾的 \0
};
```

关键字段：
- **`len`**：字符串实际已使用的长度。
- **`alloc`**：当前分配的总容量（不包括头部和结尾的空字符）。
- **`buf`**：存储字符串内容，遵循以 `\0` 结尾的惯例，以兼容部分 C 库函数。

`alloc - len` 的差值，即为 **预分配的空闲空间（free space）**。

### **3. 内存预分配策略详解**

预分配策略的核心目标是：**通过空间换时间，减少内存重分配操作**。主要包含两个机制：**空间预分配** 和 **惰性空间释放**。

#### 3.1 空间预分配（Pre-allocation）

当对一个 SDS 进行修改并需要扩展其容量时，SDS 不仅会分配修改所必需的空间，还会额外分配一定的空闲空间。

**分配规则如下：**

1.  **修改后新长度（`new_len`）小于 `SDS_MAX_PREALLOC` (默认 1MB）**：
    - **策略**：进行**加倍扩容**。新分配的容量 = `new_len * 2`。
    - **示例**：假设现有 SDS 的 `len = 10`， `alloc = 10`。现需要追加 10 字节，使 `new_len = 20`。
        - 因 `20 < 1MB`，触发加倍扩容。
        - 新 `alloc = new_len * 2 = 40`。
        - 本次分配后，`len = 20`， `alloc = 40`， 空闲空间为 20 字节。

2.  **修改后新长度（`new_len`）大于等于 `SDS_MAX_PREALLOC` (默认 1MB）**：
    - **策略**：进行**线性扩容**。每次额外多分配 `SDS_MAX_PREALLOC`（1MB）的空间。
    - **公式**：新分配的容量 = `new_len + SDS_MAX_PREALLOC`。
    - **示例**：现有 SDS 的 `len = 1MB`， `alloc = 1MB`。现需要追加 100KB 数据，使 `new_len = 1.1MB`。
        - 因 `new_len > 1MB`，触发线性扩容。
        - 新 `alloc = new_len + 1MB = 2.1MB`。
        - 分配后，空闲空间约为 1MB。

**设计 rationale**：对于较小字符串，加倍策略能带来非常显著的性能提升，且内存开销可控。对于大字符串（>1MB），加倍可能导致内存浪费过于严重（例如从 1MB 到 2MB，浪费 1MB），因此转为每次固定增加 1MB 的保守策略，在性能和内存开销间取得平衡。

#### 3.2 惰性空间释放（Lazy Free）

当需要缩短 SDS 字符串时（例如 `SETRANGE` 覆盖为空、`SDSTRIM` 修剪），SDS 不会立即回收多余的内存。

- **行为**：仅更新 `len` 字段，`alloc` 字段保持不变。被“释放”的空间仍保留在 `alloc` 中，成为空闲空间，供未来的增长操作使用。
- **显式回收**：SDS 提供了 API `sdsRemoveFreeSpace`，可真正释放未使用的内存，将 `alloc` 调整至与 `len` 相等。
- **设计 rationale**：避免缩短操作必然触发重分配。如果字符串随后再次增长，则可直接利用已有的空闲空间，效率极高。

### **4. 策略优势分析**

1.  **性能飞跃**：将追加操作从 **O(N) 的均摊时间复杂度降至接近 O(1)**。N 次连续追加仅需 O(log N) 次内存重分配，而非 N 次。
2.  **二进制安全**：SDS 依赖 `len` 而非 `\0` 判断长度，可安全存储任意二进制数据。
3.  **兼容 C 函数**：`buf` 以 `\0` 结尾，可直接使用如 `printf` 等标准库函数。
4.  **操作高效**：常数时间复杂度获取字符串长度（`strlen` 为 O(N)）。

### **5. 策略的权衡与演进**

1.  **内存占用**：预分配策略以额外的内存占用为代价。在存储海量短字符串且很少修改的场景下，可能不如 C 字符串紧凑。Redis 通过 `sdshdr5/8/16/32/64` 等不同头部结构，尽力减少小字符串的元数据开销。
2.  **内存碎片**：频繁的修改和预分配可能加剧内存碎片。Redis 的内存分配器（默认 `jemalloc`）对此进行了优化。
3.  **演进**：在 Redis 的不同版本中，SDS 的实现细节（如头部设计、类型定义）有所优化，但**空间预分配**和**惰性空间释放**的核心思想始终未变。

### **6. 示例与数据**

| 操作序列 | 旧 len | 旧 alloc | 操作 | 新 len | 新 alloc（策略应用后） | 空闲空间 | 是否重分配 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| 初始创建 | 0 | 0 | `SET key “Hello”` | 5 | 5 （初始无预分配）| 0 | 是 |
| 追加1 | 5 | 5 | `APPEND key “ World”` | 11 | 11 * 2 = **22** | 11 | 是 |
| 追加2 | 11 | 22 | `APPEND key “!”` | 12 | 22 （空间足够） | 10 | **否** |
| 大追加 | 12 | 22 | `APPEND key [1MB data]` | ~1MB+12 | (1MB+12) + 1MB = **~2MB+12** | ~1MB | 是 |
| 缩短 | ~1MB+12 | ~2MB+12 | `SETRANGE key 100 “”` | 100 | ~2MB+12 （惰性释放） | ~2MB | **否** |

从上表可以看出，通过预分配，第二次追加操作避免了内存重分配。

### **7. 总结**

Redis 的 SDS 内存预分配策略是一个经典的“空间换时间”工程实践。它通过**加倍/线性扩容**和**惰性释放**两大机制，将字符串动态修改这一高频操作的性能成本降至最低，是 Redis 实现高性能、低延迟特性的重要基石。理解这一策略，对于合理使用 Redis（例如避免大量大 Key 的频繁修改），以及设计自身的高性能数据结构，都具有重要的参考价值。
