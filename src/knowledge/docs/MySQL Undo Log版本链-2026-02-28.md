# **MySQL Undo Log 版本链技术解析**

## **1. 文档概述**

本文档旨在深入解析 MySQL InnoDB 存储引擎中 **Undo Log（回滚日志）** 的核心机制之一 —— **版本链（Version Chain）**。版本链是多版本并发控制（MVCC）得以实现的关键数据结构。通过理解版本链，可以更好地掌握 MySQL 的事务隔离性、一致性读以及行记录的隐藏逻辑。

**关键词**： `MySQL`、`InnoDB`、`Undo Log`、`版本链`、`MVCC`、`事务隔离`

## **2. 核心概念**

### **2.1 什么是 Undo Log？**
Undo Log 是 InnoDB 引擎用于保证事务**原子性（Atomicity）**和**一致性（Consistency）**的关键日志。它主要记录事务发生**之前**的数据状态。其核心作用包括：
*   **事务回滚**： 当事务执行失败或显式调用 `ROLLBACK` 时，根据 Undo Log 将数据恢复到事务开始前的状态。
*   **实现 MVCC**： 为并发事务提供一致性非锁定读，这是版本链发挥作用的主舞台。

### **2.2 什么是版本链？**
版本链是 InnoDB 中一行数据在其生命周期内，被不同事务修改所产生的**多个历史版本**，通过指针按事务提交时间顺序串联起来的**单向链表**。

*   **每行记录**都隐藏着两个关键字段：`DB_TRX_ID`（最近修改它的事务ID）和 `DB_ROLL_PTR`（回滚指针）。
*   当一个事务更新某行时，会**生成该行数据的一个新版本**，并将旧版本的数据（主要是被修改的字段值）写入 Undo Log。
*   新版本的行记录中的 `DB_ROLL_PTR` 指向其旧版本在 Undo Log 中的位置。
*   如此往复，一行数据的所有历史版本就像一条链条，通过 `DB_ROLL_PTR` 指针串联起来，这就是**版本链**。

## **3. 版本链的物理结构**

### **3.1 行记录格式与隐藏列**
InnoDB 表数据在物理上是按聚簇索引组织的。每行记录（包括聚簇索引和二级索引）都包含以下系统列：

| 列名 | 长度 | 说明 |
| :--- | :--- | :--- |
| `DB_ROW_ID` | 6 字节 | 行ID，当表无主键时自动生成。 |
| `DB_TRX_ID` | 6 字节 | **事务ID**，记录最近一次插入或更新该行的事务ID。 |
| `DB_ROLL_PTR` | 7 字节 | **回滚指针**，指向该行上一个历史版本在 Undo Log 中的记录地址。 |
| `DELETED_BIT` | 1 位 | 删除标记位，用于标记该行是否被删除。 |

`DB_ROLL_PTR` 是版本链形成的物理基础。

### **3.2 Undo Log 类型**
Undo Log 分为两种，对应不同的操作：
*   **INSERT Undo Log**： 记录插入操作，仅用于事务回滚，事务提交后可立即丢弃。
*   **UPDATE Undo Log**： 记录更新和删除操作。这是构成版本链的主体，需要为 MVCC 服务，在无活动事务依赖其版本时才能被清理。

## **4. 版本链如何工作：以 MVCC 为例**

MVCC 通过**快照读（Snapshot Read）** 实现一致性非锁定读。版本链是其核心支撑。

### **4.1 核心数据结构**
*   **Read View（读视图）**： 事务在第一次执行快照读（如 `SELECT ...` 在 `REPEATABLE READ` 级别下）时生成，它定义了当前事务的**可见性规则**。主要包含：
    *   `m_ids`： 生成 Read View 时，系统中**活跃**（未提交）的事务ID列表。
    *   `min_trx_id`： `m_ids` 中的最小值。
    *   `max_trx_id`： 生成 Read View 时，系统将分配给下一个事务的ID。
    *   `creator_trx_id`： 创建该 Read View 的事务自身的ID。

### **4.2 可见性判断算法**
当一个事务执行快照读时，对于扫描到的每一行数据，会根据其 `DB_TRX_ID` 和当前事务的 Read View，按以下规则判断应读取哪个版本：

1.  如果 `DB_TRX_ID` == `creator_trx_id`，说明这是当前事务自己修改的记录，**可见**。
2.  如果 `DB_TRX_ID` < `min_trx_id`，说明该版本在 Read View 创建前已提交，**可见**。
3.  如果 `DB_TRX_ID` >= `max_trx_id`，说明该版本在 Read View 创建后（由未来事务）生成，**不可见**。
4.  如果 `min_trx_id` <= `DB_TRX_ID` < `max_trx_id`，则检查 `DB_TRX_ID` 是否在 `m_ids` 中：
    *   如果在，说明生成该版本的事务在创建 Read View 时仍活跃，**不可见**。
    *   如果不在，说明生成该版本的事务在创建 Read View 时已提交，**可见**。

**如果当前版本不可见，则通过 `DB_ROLL_PTR` 找到上一个版本，并重复上述判断，直到找到一个可见的版本或版本链结束。**

这个过程就是 **“沿着版本链回溯”**。

## **5. 示例演示**

假设表 `t` 中有一行数据，其初始值 `id=1, value=‘A’`。

| 时间线 | 事务1 (TRX_ID=100) | 事务2 (TRX_ID=200) | 版本链状态（value列） |
| :--- | :--- | :--- | :--- |
| T1 | `BEGIN;` | `BEGIN;` | **版本V0**: value=‘A’ (TRX_ID=90, 已提交) |
| T2 | `UPDATE t SET value=‘B’ WHERE id=1;` | - | **版本V1**: value=‘B’ (TRX_ID=100, ROLL_PTR -> V0) |
| T3 | - | `SELECT * FROM t WHERE id=1;` (快照读) | 生成 Read View: `m_ids=[100, 200]`, `min=100`, `max=201` |
| | | | 事务2看到的当前行是 V1 (`DB_TRX_ID=100`)。<br>判断: `100` 在 `m_ids` 中 -> **不可见**。<br>沿指针找到 V0 (`DB_TRX_ID=90`)。<br>判断: `90` < `min_trx_id(100)` -> **可见**。<br>**结果： 读到 value=‘A’**。 |

*（事务隔离级别为 REPEATABLE READ，因此事务2的 SELECT 使用 T3 时刻生成的 Read View）*

## **6. 版本链的清理**

版本链不会无限增长。InnoDB 通过 **Purge 线程**来清理不再需要的旧版本。
*   当一个旧版本数据不再被任何活跃事务的 Read View 所需要时（即该版本对应的 `DB_TRX_ID` 小于当前系统所有 Read View 中的 `min_trx_id`），它就变得可以被清理。
*   Purge 线程会从历史链表中移除这些 Undo Log 页，并物理地删除被标记删除的行（当没有旧版本再需要它时）。

## **7. 实战意义与监控**

*   **长事务的危害**： 一个长时间未提交的事务（长事务），其 Read View 会阻止 Purge 线程清理它所能“看到”的所有旧版本，导致版本链过长，可能引发：
    *   **表空间膨胀**： Undo Log 无法及时清理，占用大量磁盘。
    *   **性能下降**： 查询时需要遍历更长的版本链，降低读性能。
*   **监控**：
    *   查询 `information_schema.innodb_trx` 查看当前运行时间较长的事务。
    *   监控 `show engine innodb status\G` 输出中 `TRANSACTIONS` 部分的 `History list length`，它大致反映了未清理的 Undo Log 数量。

## **8. 总结**

MySQL Undo Log 版本链是 InnoDB MVCC 机制的**基石**。它通过为每一行数据维护一个由回滚指针串联的历史版本链表，使得：
1.  **读操作**无需加锁即可获得一个**一致性视图**，极大提升了并发性能。
2.  **写操作**在需要时能回溯到旧版本，保证了事务的原子性和回滚能力。

理解版本链，对于深入掌握 MySQL 的事务隔离级别、排查一致性读相关问题、以及优化数据库性能（如避免长事务）都至关重要。
