# Java模块系统(JPMS)模块描述符与可见性技术文档

## 1. 概述

Java模块系统（Java Platform Module System, JPMS）是Java 9引入的核心特性，它通过模块描述符（module-info.java）实现了强封装和显式依赖管理，解决了传统类路径机制下的"JAR地狱"和缺乏可靠配置问题。

## 2. 模块描述符 (module-info.java)

### 2.1 基本结构
```java
module com.example.mymodule {
    // 模块指令
    requires java.base;           // 依赖声明
    exports com.example.api;      // 包导出
    opens com.example.internal;   // 反射开放
    provides ... with ...;        // 服务提供
    uses ...;                     // 服务消费
}
```

### 2.2 模块指令详解

#### 2.2.1 requires - 模块依赖
```java
module com.example.app {
    // 编译时和运行时都需要
    requires java.sql;
    
    // 仅编译时需要
    requires static lombok;
    
    // 仅运行时需要（Java 11+）
    requires runtime java.compiler;
    
    // 传递性依赖
    requires transitive com.example.utils;
}
```

#### 2.2.2 exports - 包导出
```java
module com.example.library {
    // 导出整个包
    exports com.example.api;
    
    // 限定性导出（qualified exports）
    exports com.example.internal 
        to com.example.app, com.example.test;
}
```

#### 2.2.3 opens - 反射开放
```java
module com.example.service {
    // 开放整个包用于反射
    opens com.example.model;
    
    // 限定性开放
    opens com.example.internal.model 
        to spring.core, hibernate.core;
    
    // 开放所有包（不推荐，破坏封装）
    open module com.example.service {
        // 指令只能放在这里，不能与其他指令混用
    }
}
```

#### 2.2.4 provides/uses - 服务机制
```java
module com.example.provider {
    provides com.example.spi.Service
        with com.example.impl.DefaultService;
}

module com.example.consumer {
    uses com.example.spi.Service;
}
```

## 3. 模块可见性规则

### 3.1 模块路径 vs 类路径
```
传统类路径: java -cp "lib/*" MainClass
模块路径:   java --module-path mods -m module/main.class
```

### 3.2 访问控制矩阵

| 访问类型 | 同一模块内 | 不同模块（已导出） | 不同模块（未导出） |
|---------|-----------|-------------------|-------------------|
| public 类 | ✅ 可访问 | ✅ 可访问 | ❌ 不可访问 |
| protected | ✅ 可访问 | ✅ 可访问（子类）| ❌ 不可访问 |
| 包私有   | ✅ 可访问 | ❌ 不可访问 | ❌ 不可访问 |
| private  | ✅ 可访问 | ❌ 不可访问 | ❌ 不可访问 |

### 3.3 反射访问规则
```java
// 传统方式（Java 8及以前）
Class<?> clazz = Class.forName("com.example.internal.PrivateClass");
Field field = clazz.getDeclaredField("secret");
field.setAccessible(true);  // 可以访问

// 模块化后（未open的情况下）
// 抛出IllegalAccessException: module does not open ...
```

## 4. 模块描述符示例

### 4.1 简单库模块
```java
module com.example.utils {
    requires transitive java.logging;
    
    exports com.example.utils;
    exports com.example.utils.internal 
        to com.example.app;
    
    opens com.example.utils.model 
        to com.fasterxml.jackson.databind;
}
```

### 4.2 主应用模块
```java
module com.example.app {
    requires com.example.utils;
    requires java.sql;
    requires static com.example.annotations;
    
    exports com.example.app.api;
    
    opens com.example.app.model 
        to spring.core;
    
    provides com.example.spi.Plugin
        with com.example.app.plugin.MainPlugin;
    
    uses com.example.spi.Extension;
}
```

## 5. 编译与运行

### 5.1 编译模块
```bash
# 编译单个模块
javac -d out/module \
      --module-path deps \
      --module-source-path src \
      src/module-info.java src/com/example/**/*.java

# 编译多个模块
javac -d out \
      --module-path deps \
      --module-source-path src \
      -m module1,module2
```

### 5.2 运行模块
```bash
# 运行模块化应用
java --module-path out:deps \
     -m com.example.app/com.example.app.Main

# 添加VM参数
java --module-path mods \
     --add-opens module/package=target-module \
     --add-exports module/package=target-module \
     -m app/main.class
```

### 5.3 创建模块JAR
```bash
# 创建普通模块JAR
jar --create \
    --file=myModule.jar \
    --module-version=1.0 \
    -C out/module .

# 创建可执行JAR（指定主类）
jar --create \
    --file=myApp.jar \
    --main-class=com.example.app.Main \
    -C out/app .
```

## 6. 迁移策略与最佳实践

### 6.1 渐进式迁移
1. **自动模块**：将非模块化JAR放在module-path，自动转换为模块
   ```bash
   java --module-path legacy.jar:mods -m my.module
   ```

2. **未命名模块**：类路径中的JAR作为未命名模块，可访问所有模块

### 6.2 模块设计原则
1. **强封装优先**：默认不导出，按需开放
2. **最小化API**：使用限定性导出缩小API表面
3. **避免循环依赖**：使用服务机制解耦
4. **分层设计**：
   ```
   ├── 应用层 (app)        - 使用模块
   ├── 业务层 (service)    - 导出接口
   ├── 基础设施层 (infra)  - 传递依赖
   └── 领域层 (domain)     - 核心模型
   ```

### 6.3 常见问题解决
```java
// 问题1：需要访问内部API
// 解决方案：使用--add-opens或--add-exports
java --add-opens source.module/pkg=target.module

// 问题2：反射框架需要访问私有字段
// 解决方案：在模块描述符中添加opens指令
opens com.example.model to hibernate.core;

// 问题3：测试需要访问内部类
// 解决方案：测试依赖或使用测试专用模块
exports com.example.internal to com.example.test;
```

## 7. 工具支持

### 7.1 JDK工具增强
- `jdeps`: 分析模块依赖
- `jlink`: 创建自定义运行时映像
- `jmod`: 处理JMOD文件

### 7.2 构建工具
- Maven: `maven-compiler-plugin` 3.6+
- Gradle: `java` 和 `java-library` 插件
- IDE: IntelliJ IDEA, Eclipse 全面支持

## 8. 总结

Java模块系统通过模块描述符实现了：
1. **可靠的配置**：显式声明依赖，启动时验证
2. **强封装**：精确控制包的可见性
3. **可扩展性**：支持服务提供者机制
4. **安全性**：限制反射访问，保护内部API
5. **性能优化**：类加载优化，支持jlink创建最小运行时

模块系统是Java平台现代化的关键一步，虽然引入了一定的学习成本，但为大型应用的可维护性、安全性和性能提供了坚实基础。
