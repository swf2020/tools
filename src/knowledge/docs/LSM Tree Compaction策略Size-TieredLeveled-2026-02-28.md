## LSM Tree Compaction 策略详解：Size-Tiered 与 Leveled

### 1. 概述

日志结构合并树（Log-Structured Merge-Tree, LSM Tree）是一种高性能的写优化数据结构，广泛应用于现代数据库和存储系统（如Cassandra, RocksDB, HBase等）。其核心思想是将随机写转换为顺序写，从而大幅提升写入吞吐量。

Compaction（压缩/合并）是LSM Tree最关键的背景操作之一，主要负责：
*   **合并多个有序数据文件（SSTable）**，消除重复和已删除的数据。
*   **维持读性能**，通过减少需要查找的文件数量。
*   **回收存储空间**。

不同的Compaction策略在**写放大、读放大、空间放大**这三个核心权衡上表现迥异。**Size-Tiered** 和 **Leveled** 是目前两种最主要和基础的策略。

### 2. Size-Tiered Compaction Strategy (STCS)

#### 2.1 核心思想
“大小分层”策略。其核心是将**大小近似**的SSTable合并成更大的一个SSTable。数据在“层级”上不是按Key范围严格划分，而是按文件大小自然形成的“代”。

#### 2.2 工作流程
1.  **内存表（MemTable）** 写满后，刷写到磁盘，形成第一个 **L0 SSTable**（大小约为 `MemTableSize`）。
2.  当数量达到一定阈值（例如4个）大小近似的L0 SSTable时，将它们合并成一个更大的 **L1 SSTable**。
3.  当又积累到足够数量（例如4个）大小近似的L1 SSTable时，将它们合并成一个更大的 **L2 SSTable**。
4.  如此递归向上，越往上层，SSTable的文件尺寸越大，数量通常越少。

**关键参数**：`tiering_compaction_threshold` (通常为4)。即每层需要凑齐N个大小相近的文件才触发合并。

#### 2.3 优缺点
**优点**：
*   **写入放大低**：合并操作发生在大小相近的文件之间，每次合并的数据量相对可控，写放大通常被认为是 O(log N)。
*   **写入吞吐高**：由于合并频率相对较低且策略简单，对写入流程的干扰较小，非常适用于写密集型场景。
*   **实现简单**。

**缺点**：
*   **读放大高**：一个Key可能存在于同一层的多个SSTable中，读取时需要查询所有可能包含该Key的文件，即使使用Bloom Filter，在最坏情况下仍需检查多个文件。
*   **空间放大高**：在合并触发前，同一份数据的多个版本（旧值和新值）可能共存于多个SSTable中，导致存储空间占用较大。峰值时可能达到原始数据的数倍。
*   **数据新鲜度**：旧数据可能长时间存在于未合并的SSTable中，范围查询效率较低。

### 3. Leveled Compaction Strategy (LCS)

#### 3.1 核心思想
“严格分层”策略。其核心是**将数据按Key范围划分到不同的“层级”（Level）中，并保证除L0外的每一层内，SSTable的Key范围互不重叠**。这使得每一层在逻辑上都是一个全局有序的大SSTable。

#### 3.2 工作流程
1.  MemTable刷盘形成 **L0 SSTable**。L0允许文件间Key范围重叠。
2.  L0的文件积累到一定数量后，会与 **L1** 中Key范围有交集的SSTable进行合并。合并结果输出到L1。
3.  **从L1开始及以后，每一层（Lk）都有固定的大小上限**（例如，L1是100MB，L2是1GB，L3是10GB……呈指数增长）。
4.  当Lk层大小超过上限时，会从Lk中选择一个SSTable，与Lk+1层中所有Key范围与之重叠的SSTable进行合并。合并结果输出到Lk+1层，并保证Lk+1层在合并后依然满足“层内无重叠”的条件。
5.  合并是逐层、小范围、增量式进行的。

**关键参数**：`level0_file_num_compaction_trigger` (触发L0->L1合并的文件数)，以及各层的目标大小。

#### 3.3 优缺点
**优点**：
*   **读放大低**：对于点查询，除L0外，每一层最多只需要检查一个SSTable（因为Key范围不重叠）。结合Bloom Filter，读性能非常稳定且高效。
*   **空间放大低**：由于层内无重叠，非L0层中基本没有冗余数据。空间放大通常可以控制在1.1倍左右，远优于STCS。
*   **范围查询效率高**：可以直接在每一层的少数几个有序文件中进行高效归并。

**缺点**：
*   **写入放大高**：数据从高层合并到更高层时，可能会被反复读写。在最坏情况下，写放大可以达到 O(Levels * (Fanout Ratio))，例如10-20倍，甚至更高。这会影响SSD寿命和写入吞吐。
*   **磁盘I/O压力更持续**：由于是增量式、小范围的合并，对I/O资源的占用更平缓但更持续。

### 4. 核心对比与选型指南

| 特性维度 | Size-Tiered (STCS) | Leveled (LCS) |
| :--- | :--- | :--- |
| **核心原理** | 合并大小相近的文件 | 分层且层内Key不重叠 |
| **写入放大** | **低** | **高** |
| **读放大** | **高**（需查同层多个文件） | **低**（每层最多查一个文件） |
| **空间放大** | **高**（合并前多版本共存） | **低**（约1.1倍） |
| **写吞吐** | **更高** | 相对较低 |
| **读延迟** | 不稳定，较高 | **稳定且低** |
| **范围查询** | 效率较低 | **效率高** |
| **实现复杂度** | 简单 | 复杂 |
| **典型应用** | Cassandra (默认)， ScyllaDB | RocksDB (默认)， LevelDB， HBase |

**选型建议**：

*   **选择 Size-Tiered (STCS) 如果**：
    *   应用是**极端的写密集型**（如时序数据、日志摄入）。
    *   对写入成本和吞吐有极高要求。
    *   可以接受相对较高且不稳定的读取延迟。
    *   存储空间相对充裕。

*   **选择 Leveled (LCS) 如果**：
    *   应用是**读密集型**或需要**混合负载**。
    *   **点查询和范围查询的延迟与吞吐**是关键指标。
    *   存储空间有限或成本敏感。
    *   愿意为更好的读性能牺牲一部分写入性能。

### 5. 总结与演进

Size-Tiered和Leveled策略代表了LSM Tree Compaction设计空间的两个极端：**写优化** vs **读优化**。现代数据库系统通常不会只提供这两种纯策略，而是：
1.  **提供可配置策略**：允许用户根据负载特征选择。
2.  **提供混合或变种策略**：
    *   **Time Window Compaction**：基于时间的STCS变种，常用于时序数据。
    *   **Tiered+Leveled Hybrid**：在L0使用Tiered策略缓冲写入，在深层使用Leveled策略优化读。
    *   **Leveled-N**：在Leveled基础上，允许每层有少量（N个）重叠文件，以平衡写放大。
    *   **FIFO Compaction**：最简单的策略，直接按时间顺序删除旧文件，适用于纯临时数据。

理解这两种基础策略的机制与权衡，是合理设计、调优和使用任何基于LSM Tree存储系统的基石。在实际系统中，应根据数据模型、访问模式、硬件资源（尤其是SSD耐久度）进行细致的测试与选择。
