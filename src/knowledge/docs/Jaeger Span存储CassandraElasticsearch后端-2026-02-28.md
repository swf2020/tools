# **Jaeger Span 存储技术文档：Cassandra 与 Elasticsearch 后端**

## **1. 概述**

Jaeger 是一款开源的分布式追踪系统，用于监控和诊断基于微服务的分布式系统。收集到的追踪数据（Traces）由多个跨度（Spans）组成，这些数据需要被持久化存储以供查询和分析。

Jaeger 默认支持多种存储后端，其中 **Cassandra** 和 **Elasticsearch** 是两种最常用、官方支持度最高的生产级可扩展存储方案。本文档旨在详细阐述这两种后端存储的架构、配置、优缺点以及选型建议。

## **2. 核心存储架构**

Jaeger 的存储设计遵循可插拔架构。收集器（Collector）从代理（Agent）接收 spans，经过处理（如索引生成）后，通过统一的存储接口写入配置的后端。

### **2.1 数据模型与存储需求**
一条完整的 Trace 在存储层被拆解为两部分：
1.  **Span 数据**：存储 Trace 的核心细节，包括操作名、时间戳、标签（Tags）、日志（Logs）等。查询时需按 Trace ID 高效检索。
2.  **索引数据**：为支持灵活的搜索（如根据服务名、操作名、标签进行查询），需要建立额外的索引。这些索引通常以“服务-操作-时间”或“标签键值对”等维度组织。

Cassandra 和 Elasticsearch 以不同的方式满足这两类需求。

## **3. Cassandra 后端**

### **3.1 架构设计**
Jaeger 使用 **纯 Cassandra** 来存储所有数据（Span数据和索引数据）。它通过精心设计的表结构和主键定义来实现高效的查询。
*   **Span 存储**：`traces` 表。主键为 `(bucket, timestamp)`，其中 `bucket` 是基于 Trace ID 和时间戳计算出的一个时间桶（例如按天分区），用于将数据分散到不同分区，避免热点。
*   **索引存储**：
    *   `service_name_index` / `service_operation_index`：用于按服务和操作查找 Trace ID。
    *   `tags_index`：用于按标签查找 Trace ID。
    *   `duration_index`：用于按耗时范围查找 Trace ID。
    *   索引表的主键通常包含 `(service, bucket)` 等，以实现高效的范围查询。

### **3.2 特点**
*   **写入优化**：为高吞吐量写入场景设计。索引写入与Span写入同步进行（通过批处理），保证查询能力。
*   **最终一致性**：Cassandra 的架构决定了其查询通常是最终一致性的。刚写入的 Span 可能无法立即在索引查询中看到。
*   **存储效率**：相比 Elasticsearch，通常具有更高的原始数据存储压缩率。
*   **运维复杂**：需要管理单独的 Cassandra 集群，包括容量规划、压缩、修复等运维工作。
*   **查询灵活性有限**：索引查询依赖于预先定义好的表结构。不支持 Elasticsearch 那样丰富的全文搜索或复杂的多条件聚合分析。

### **3.3 配置要点**
```yaml
# Jaeger Collector/Query 服务配置示例 (环境变量)
SPAN_STORAGE_TYPE=cassandra
CASSANDRA_SERVERS=jaeger-db-host1,jaeger-db-host2
CASSANDRA_KEYSPACE=jaeger_v1_test
CASSANDRA_CONSISTENCY=QUORUM # 读写一致性级别
# 可配置连接池、超时、认证等参数
```
初始化 Keyspace 和 Schema 需要使用 Jaeger 提供的 `hotrod` 镜像或脚本工具。

## **4. Elasticsearch 后端**

### **4.1 架构设计**
Jaeger 使用 **纯 Elasticsearch** 作为存储后端。
*   **Span 与索引的统一存储**：所有数据（Span 详情和索引）都以 JSON 文档形式存储在 Elasticsearch 的索引中。一个 Span 文档包含其所有详细信息。
*   **利用 ES 原生能力**：
    *   **倒排索引**：自动为文档的所有字段（服务名、标签等）建立索引，支持极其灵活和强大的查询。
    *   **全文搜索**：支持对标签值、日志字段进行模糊匹配和分词查询。
    *   **聚合分析**：支持基于字段的复杂聚合操作（如统计某个错误的出现频率、绘制耗时百分位图）。

### **4.2 特点**
*   **查询能力强大**：提供比 Cassandra 丰富得多的查询选项和聚合分析能力，便于深度钻取问题。
*   **开发体验好**：可直接使用 Kibana 对 Jaeger 数据进行可视化分析和探索。
*   **写入开销**：为每个字段建立索引会带来更高的 CPU、内存消耗和磁盘开销（相比原始数据膨胀）。
*   **大规模运维挑战**：Elasticsearch 集群本身的运维（分片管理、映射模板、版本升级、性能调优）具有一定复杂性。
*   **数据新鲜度**：取决于 Elasticsearch 的刷新间隔（通常为1秒），近实时可查。

### **4.3 配置要点**
```yaml
# Jaeger Collector/Query 服务配置示例 (环境变量)
SPAN_STORAGE_TYPE=elasticsearch
ES_SERVER_URLS=http://es-host1:9200,http://es-host2:9200
ES_INDEX_PREFIX=jaeger- # 索引名前缀，如 jaeger-span-2024-10-22
ES_TAGS_AS_FIELDS_ALL=true # 将所有标签存储为独立的 ES 字段，极大提升标签查询性能
ES_VERSION=7 # 指定 ES 主版本
# 可配置认证、嗅探、批量写入参数等
```
Jaeger 会自动创建和管理按日期滚动的索引（例如 `jaeger-span-2024-10-22`）。

## **5. 对比与选型建议**

| 特性维度 | **Cassandra** | **Elasticsearch** |
| :--- | :--- | :--- |
| **核心优势** | 高吞吐写入，成本效益高，存储紧凑 | 强大的即席查询、全文搜索和聚合分析能力 |
| **查询灵活性** | **低**：仅限于预定义索引的查询 | **高**：支持任意字段的组合、模糊、范围查询和复杂聚合 |
| **写入性能** | **极高**：为写入密集型优化，线性扩展性好 | **高**：但为支持灵活查询，索引开销会消耗更多资源 |
| **存储开销** | **较低**：压缩效率高，原始数据存储 | **较高**：倒排索引导致数据膨胀，通常建议使用更高性能的 SSD |
| **运维复杂度** | **中高**：需管理 Cassandra 集群（修复、压缩） | **中高**：需管理 ES 集群（分片、映射、版本） |
| **生态系统** | 相对简单，监控工具成熟（如 Prometheus） | 与 **Kibana** 深度集成，提供开箱即用的可视化分析 |
| **典型适用场景** | 超大规模追踪数据采集，关注写入成本和吞吐，查询模式固定（如仅按TraceID或服务名查找）。 | 需要频繁进行问题调查、根因分析，依赖灵活的标签查询、关键词搜索或业务指标聚合。开发/调试阶段。 |

### **5.1 选型建议**
*   **选择 Cassandra 如果**：
    *   你的主要需求是**稳定、高效地海量接收和存储追踪数据**，对存储成本敏感。
    *   查询模式相对简单固定，大部分查询通过 Trace ID 或服务/操作名即可完成。
    *   团队已有 Cassandra 的运维经验或基础设施。

*   **选择 Elasticsearch 如果**：
    *   **强大的查询和诊断能力是你的首要需求**。你需要经常根据各种标签（如 `http.status_code=500`、`error=true`）或业务属性进行过滤和聚合。
    *   你希望团队能够使用 **Kibana** 直接探索和分析追踪数据，进行自定义可视化。
    *   团队已有 Elasticsearch 技术栈和经验。

## **6. 运维与监控**

*   **容量规划**：无论选择哪种，都需要根据 Span 量/日、保留策略（TTL）来规划存储容量和集群规模。
*   **数据生命周期管理**：
    *   **Cassandra**：Jaeger 内置支持基于 TTL 的自动过期。需设置合理的 `cassandra.ttl`。
    *   **Elasticsearch**：可使用 ES 原生的 **索引生命周期管理（ILM）** 策略，自动滚动、收缩、删除旧索引。这是推荐的实践。
*   **监控**：
    *   监控存储后端的健康状态、读写延迟、错误率、磁盘使用率。
    *   监控 Jaeger Collector 的队列长度和写入错误。
*   **高可用**：为生产环境配置多副本。Cassandra 依赖其内置的多副本机制；Elasticsearch 通过分片副本实现。

## **7. 总结**

Cassandra 和 Elasticsearch 都是 Jaeger 优秀的、经过生产验证的存储后端，但它们服务于略有不同的侧重点。**Cassandra 是“存储引擎”**，追求写入效率和成本；**Elasticsearch 是“搜索引擎”**，追求查询的灵活性和深度。在选型时，应紧密结合团队的技术栈、运维能力，以及最重要的——对追踪数据的使用模式（是重在归档，还是重在交互式分析）来做出决策。

对于追求极致查询体验和问题诊断效率的团队，**Elasticsearch 正日益成为默认的推荐选择**。
