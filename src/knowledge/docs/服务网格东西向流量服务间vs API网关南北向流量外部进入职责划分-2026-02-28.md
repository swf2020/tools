# **服务网格东西向流量与API网关南北向流量职责划分技术文档**

**文档版本：** 1.0  
**发布日期：** 2023-10-27  
**关键词：** 服务网格， API网关， 东西向流量， 南北向流量， 微服务， 云原生

---

## **1. 概述**

在现代微服务与云原生架构中，流量管理被清晰地划分为两个核心平面：**服务间通信（东西向流量）** 和 **外部客户端到服务的通信（南北向流量）**。这两种流量模式具有不同的关注点、挑战和解决方案。本技术文档旨在明确 **服务网格（Service Mesh）** 与 **API网关（API Gateway）** 在这两个领域中的核心职责、技术边界以及最佳实践，为架构设计与技术选型提供清晰指导。

**核心概念定义：**
*   **东西向流量 (East-West Traffic)：** 指在数据中心或集群内部，服务实例之间的通信流量。例如，`订单服务`调用`支付服务`，`用户服务`调用`数据库服务`。
*   **南北向流量 (North-South Traffic)：** 指从外部网络（如互联网、移动端）进入数据中心或集群的入口流量，以及从内部服务流出到外部的响应流量。例如，用户通过浏览器访问前端应用，移动APP调用后端API。
*   **服务网格 (Service Mesh)：** 一个专注于处理服务间通信的基础设施层，提供可靠、安全和可观测的网络代理。
*   **API网关 (API Gateway)：** 作为系统唯一的入口点，负责处理所有外部客户端的请求，是面向业务API的管理和路由枢纽。

## **2. 职责边界对比**

为了清晰划分，以下表格从多个维度对比了两者的核心职责：

| 维度 | **服务网格 (处理东西向流量)** | **API网关 (处理南北向流量)** |
| :--- | :--- | :--- |
| **核心定位** | **服务间通信基础设施层**，是服务的“网络内核”。 | **外部流量统一入口**，是系统的“边界守卫和前台”。 |
| **核心用户** | **服务开发者、运维人员**。关注服务如何稳定、安全地互相调用。 | **API消费者、合作伙伴、前端/移动端开发者**。关注如何高效、安全地访问业务能力。 |
| **核心职责** | **服务治理**：服务发现、负载均衡、熔断、重试、超时、故障注入。 <br> **安全通信**：服务身份认证、自动mTLS加密、细粒度服务间授权（RBAC）。 <br> **可观测性**：服务级指标（如延迟、错误率、RPS）、分布式追踪、服务依赖拓扑。 <br> **流量管理**：金丝雀发布、蓝绿部署、基于内容/权重的路由（服务间）。 | **API管理**：API聚合、协议转换（REST/gRPC/GraphQL）、请求/响应转换、API版本管理。 <br> **边缘功能**：用户认证鉴权（OAuth2/JWT/API Key）、限流/防刷、缓存、WAF/DDOS防护。 <br> **业务路由**：将外部请求路由到对应的后端服务，通常基于HTTP路径、域名、Header。 <br> **计量计费**：API调用次数计量，为商业化提供支持。 |
| **部署位置** | 每个服务实例旁，作为**Sidecar代理**（如Envoy），与业务服务同生命周期。 | 集群边缘，作为**独立部署的入口点**（如Kubernetes Ingress前方），通常有高可用集群。 |
| **配置维度** | 以 **服务/工作负载** 为维度。策略应用于“从服务A到服务B”的通信。 | 以 **API/路由** 为维度。策略应用于“对`/api/v1/orders`路径的GET请求”。 |
| **网络层级** | 主要工作在L4（TCP）和L7（HTTP/gRPC），关注**传输层和微服务协议层**。 | 主要工作在L7（HTTP/HTTPS/WebSocket），关注**应用层和业务API协议**。 |

## **3. 架构示意图与协作模式**

### **3.1 典型架构图**

```
+---------------------+      +---------------------------------------+
|   外部客户端        |      |          Kubernetes 集群              |
| (Web/App/Partner)   |      |                                       |
+---------------------+      |  +-------------+    +-------------+   |
          |                  |  |  服务网格    |    |  API网关     |   |
          | 南北向流量       |  |  (数据平面)  |    |   Pod/实例   |   |
          v                  |  | +---------+  |    | +---------+  |   |
+----------------------+     |  | | Sidecar |<----->| |  代理   |  |   |
|  负载均衡器          |     |  | +---------+  |    | +---------+  |   |
| (Cloud LB / F5)     |     |  |       ^       |    |       ^       |   |
+----------------------+     |  |       |       |    |       |       |   |
          |                  |  |       |东西向流量  |       |南北向入口|
          |                  |  |       v       |    |       v       |   |
          +------------------->| +---------+    |    | +---------+  |   |
                             | | | 业务服务A |    |    | 路由规则   |  |   |
                             | | +---------+    |    | +---------+  |   |
                             | |     | 调用      |    |       |       |   |
                             | |     v          |    |       |       |   |
                             | | +---------+    |    | +---------+  |   |
                             | | | 业务服务B |<------->| 后端服务群 |  |   |
                             | | +---------+    |    | +---------+  |   |
                             | +-------------+    +-------------+   |
                             +---------------------------------------+
                                          ^
                                          |
                                    +-------------+
                                    | 控制平面    |
                                    | (如Istiod) |
                                    +-------------+
```

**流程说明：**
1. 外部请求通过**负载均衡器**抵达**API网关**。
2. API网关执行身份验证、限流等边缘策略后，根据路由规则将请求转发至对应的**后端服务**（如业务服务A）。
3. 业务服务A需要调用业务服务B，此**服务间调用**由**服务网格**接管。
4. 服务网格的Sidecar代理自动完成服务发现、负载均衡、mTLS加密、重试等治理动作，将请求发往业务服务B的Sidecar。
5. 所有流量（东西向和经过网关的南北向）的指标、日志和追踪信息被收集至可观测性平台。

### **3.2 互补与协作**

服务网格与API网关并非替代关系，而是**互补与协作**关系，共同构成完整的云原生流量管理体系。

1.  **清晰的边界**：API网关是“对外门户”，服务网格是“内部高速公路”。网关不应处理大量的服务间交叉调用。
2.  **安全纵深**：API网关负责第一道外部安全（用户身份），服务网格负责更深层的内部零信任安全（服务身份）。外部用户通过网关认证后，其请求在服务网格内部流转时仍受服务间策略保护。
3.  **关注点分离**：网关团队（或API管理团队）专注于API设计、用户体验和商业条款；服务开发团队专注于通过服务网格实现敏捷、可靠的服务间通信。

## **4. 使用场景与决策建议**

### **何时使用服务网格？**
*   您有大量服务，且服务间通信复杂，需要统一的治理、可观测性和安全策略。
*   您正在实施**零信任安全模型**，要求服务间通信必须经过认证和加密。
*   您需要频繁进行精细的**灰度发布**（如按请求头、用户百分比）和故障恢复演练。
*   您的团队希望将通信逻辑（重试、熔断）从业务代码中剥离，实现**非侵入式治理**。

### **何时使用API网关？**
*   您需要为外部客户端（Web、移动端、合作伙伴）提供一个**统一、稳定的API入口**。
*   您需要管理**API的生命周期**（发布、版本、下线）和**开发者门户**。
*   您需要对API访问进行**计量、计费和限流**。
*   您需要进行**协议转换**（如将内部gRPC服务暴露为RESTful API）。
*   您需要集中实施针对外部流量的**安全防护**（如防爬虫、防注入）。

### **典型部署模式**
*   **模式一（推荐）：API网关 + 服务网格**
    *   **描述**：在集群入口部署API网关处理所有南北向流量；集群内部所有服务接入服务网格处理东西向流量。
    *   **优点**：职责清晰，能力完备，安全性最高，是生产环境的标配。
*   **模式二：仅API网关（处理简单场景）**
    *   **描述**：服务数量少，复杂度低，服务间治理需求不强。API网关兼顾简单路由和边缘功能。
    *   **优点**：架构简单，运维成本低。适用于微服务初期或中小型应用。
*   **模式三：服务网格承担部分南北向流量（Ingress Gateway）**
    *   **描述**：使用服务网格提供的**Ingress Gateway**组件替代或简化传统API网关的流量路由功能，但仍需一个轻量级入口或保留API网关用于高级API管理。
    *   **优点**：可以利用服务网格的同一套策略和可观测性模型管理入口流量。**注意**：它通常缺乏成熟的API管理、门户和商业化功能。

## **5. 总结**

在云原生架构中，明确 **“API网关管南北，服务网格管东西”** 的职责划分是构建健壮、安全和可维护系统的关键。

*   **API网关** 是业务能力的**外交官**，专注于对外API的呈现、管理和保护。
*   **服务网格** 是微服务世界的**神经系统**，专注于内部服务间通信的可靠性、安全性和可观测性。

将两者结合使用，可以实现从边缘到核心的全栈流量管理，既能满足外部用户的多样化需求，又能保障内部架构的敏捷与稳定。在技术选型时，应根据组织的具体规模、复杂度、安全要求和团队技能，选择适合的模式与产品（如 Istio/Linkerd 用于服务网格，Kong/APISIX/Tyk 用于API网关）。
