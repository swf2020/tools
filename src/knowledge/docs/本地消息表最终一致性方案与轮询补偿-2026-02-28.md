# 本地消息表最终一致性方案与轮询补偿技术文档

## 1. 概述

### 1.1 方案简介
本地消息表方案是一种分布式事务最终一致性解决方案，通过在业务数据库中创建消息表记录事务状态，配合异步轮询补偿机制，确保跨服务事务的最终一致性。

### 1.2 核心思想
- **事务拆分**：将分布式事务拆分为多个本地事务
- **消息持久化**：在执行业务操作时同步记录消息状态
- **异步补偿**：通过定时任务轮询补偿未完成的事务
- **最终一致**：保证系统在有限时间内达到一致性状态

## 3. 方案实现

### 3.1 事务执行流程

```java
/**
 * 本地消息表事务执行器
 */
@Service
public class LocalMessageTransactionExecutor {
    
    @Autowired
    private MessageSender messageSender;
    
    @Autowired
    private LocalMessageService localMessageService;
    
    @Transactional(rollbackFor = Exception.class)
    public void executeWithLocalMessage(
            Runnable businessAction, 
            MessageInfo messageInfo) {
        
        // 1. 执行业务操作
        businessAction.run();
        
        // 2. 保存本地消息（与业务操作在同一个事务中）
        LocalMessage message = LocalMessage.builder()
            .messageId(UUID.randomUUID().toString())
            .businessKey(messageInfo.getBusinessKey())
            .topic(messageInfo.getTopic())
            .content(messageInfo.getContent())
            .status(MessageStatus.PENDING.getCode())
            .build();
        localMessageService.save(message);
        
        // 3. 异步发送消息（事务提交后执行）
        TransactionSynchronizationManager.registerSynchronization(
            new TransactionSynchronization() {
                @Override
                public void afterCommit() {
                    messageSender.sendAsync(message);
                }
            }
        );
    }
}
```

### 3.2 轮询补偿服务

```java
/**
 * 消息补偿调度器
 */
@Component
public class MessageCompensator {
    
    @Autowired
    private LocalMessageService localMessageService;
    
    @Autowired
    private MessageSender messageSender;
    
    /**
     * 补偿待发送消息
     */
    @Scheduled(fixedDelay = 5000) // 每5秒执行一次
    public void compensatePendingMessages() {
        List<LocalMessage> pendingMessages = localMessageService
            .findByStatusAndRetry(
                MessageStatus.PENDING.getCode(),
                3 // 最大重试次数
            );
        
        for (LocalMessage message : pendingMessages) {
            try {
                messageSender.send(message);
                localMessageService.updateStatus(
                    message.getId(), 
                    MessageStatus.SENT.getCode()
                );
            } catch (Exception e) {
                handleSendFailure(message, e);
            }
        }
    }
    
    /**
     * 补偿未确认消息
     */
    @Scheduled(fixedDelay = 30000) // 每30秒执行一次
    public void compensateUnconfirmedMessages() {
        List<LocalMessage> unconfirmedMessages = localMessageService
            .findSentButUnconfirmed(24 * 60); // 超过24小时未确认
        
        for (LocalMessage message : unconfirmedMessages) {
            // 1. 查询下游服务状态
            boolean confirmed = queryDownstreamStatus(message);
            
            // 2. 根据查询结果更新状态或重发
            if (confirmed) {
                localMessageService.updateStatus(
                    message.getId(), 
                    MessageStatus.CONFIRMED.getCode()
                );
            } else if (message.getRetryCount() < message.getMaxRetry()) {
                messageSender.resend(message);
            }
        }
    }
    
    private void handleSendFailure(LocalMessage message, Exception e) {
        int retryCount = message.getRetryCount() + 1;
        
        if (retryCount >= message.getMaxRetry()) {
            // 达到最大重试次数，标记为失败
            localMessageService.updateStatus(
                message.getId(), 
                MessageStatus.FAILED.getCode()
            );
            // 发送告警通知
            alertService.sendAlert(message, e);
        } else {
            // 更新重试信息
            localMessageService.updateRetryInfo(
                message.getId(),
                retryCount,
                calculateNextRetryTime(retryCount)
            );
        }
    }
}
```

### 3.3 幂等性处理

```java
/**
 * 消息消费者幂等处理器
 */
@Component
public class IdempotentMessageConsumer {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    private static final String PROCESSED_KEY_PREFIX = "msg:processed:";
    private static final long EXPIRE_TIME = 24 * 60 * 60; // 24小时
    
    /**
     * 判断消息是否已处理（幂等性检查）
     */
    public boolean isProcessed(String messageId) {
        String key = PROCESSED_KEY_PREFIX + messageId;
        Boolean exists = redisTemplate.hasKey(key);
        return Boolean.TRUE.equals(exists);
    }
    
    /**
     * 标记消息已处理
     */
    public void markAsProcessed(String messageId) {
        String key = PROCESSED_KEY_PREFIX + messageId;
        redisTemplate.opsForValue().set(key, "1", EXPIRE_TIME, TimeUnit.SECONDS);
    }
    
    /**
     * 幂等消费消息
     */
    @Transactional(rollbackFor = Exception.class)
    public void consumeIdempotently(Message message, Consumer<Message> businessLogic) {
        // 1. 幂等性检查
        if (isProcessed(message.getMessageId())) {
            log.info("消息已处理，跳过重复消费: {}", message.getMessageId());
            return;
        }
        
        // 2. 执行业务逻辑
        businessLogic.accept(message);
        
        // 3. 标记为已处理
        markAsProcessed(message.getMessageId());
        
        // 4. 发送确认消息
        sendConfirmation(message);
    }
}
```

## 4. 配置与优化

### 4.1 参数配置示例
```yaml
# application.yml
message:
  compensation:
    # 轮询配置
    pending-message-interval: 5000      # 待发送消息轮询间隔(ms)
    unconfirmed-message-interval: 30000 # 未确认消息轮询间隔(ms)
    batch-size: 100                     # 每次处理批次大小
    
    # 重试策略
    max-retry-count: 3                  # 最大重试次数
    retry-backoff:
      initial-interval: 1000            # 初始重试间隔(ms)
      multiplier: 2.0                   # 重试间隔倍数
      max-interval: 60000               # 最大重试间隔(ms)
    
    # 超时配置
    process-timeout: 30000              # 消息处理超时时间(ms)
    confirmation-timeout: 86400000      # 确认超时时间(ms) - 24小时
```

### 4.2 性能优化建议

1. **数据库优化**
   - 为状态字段和创建时间添加复合索引
   - 定期归档历史消息数据
   - 使用分库分表处理海量消息数据

2. **轮询优化**
   - 使用分布式锁避免重复补偿
   - 实现增量轮询（记录上次处理位置）
   - 根据负载动态调整轮询频率

3. **内存优化**
   - 限制单次轮询处理的消息数量
   - 实现消息批量发送机制
   - 使用连接池管理数据库连接

## 5. 监控与告警

### 5.1 关键监控指标

```java
/**
 * 消息补偿监控
 */
@Component
public class MessageCompensationMonitor {
    
    @Autowired
    private MeterRegistry meterRegistry;
    
    private final Counter pendingMessageCounter;
    private final Timer compensationTimer;
    private final Counter failureCounter;
    
    public MessageCompensationMonitor() {
        this.pendingMessageCounter = Counter.builder("message.pending.count")
            .description("待处理消息数量")
            .register(meterRegistry);
        
        this.compensationTimer = Timer.builder("message.compensation.time")
            .description("补偿处理耗时")
            .register(meterRegistry);
        
        this.failureCounter = Counter.builder("message.compensation.failure")
            .description("补偿失败次数")
            .register(meterRegistry);
    }
    
    public void recordCompensation(long duration, boolean success) {
        compensationTimer.record(duration, TimeUnit.MILLISECONDS);
        if (!success) {
            failureCounter.increment();
        }
    }
}
```

## 6. 异常处理与容错

### 6.1 常见异常场景

1. **消息发送失败**
   - 网络异常
   - 消息队列服务不可用
   - 消息格式错误

2. **业务处理失败**
   - 下游服务不可用
   - 数据库连接异常
   - 业务逻辑错误

3. **状态不一致**
   - 消息已发送但未确认
   - 重复消费
   - 消息丢失

### 6.2 容错策略

```java
/**
 * 容错处理器
 */
@Component
public class FaultToleranceHandler {
    
    /**
     * 降级策略：当消息队列不可用时
     */
    public void handleMQUnavailable(LocalMessage message) {
        // 1. 记录日志
        log.error("消息队列不可用，消息转入降级处理: {}", message.getMessageId());
        
        // 2. 将消息标记为等待重试
        message.setStatus(MessageStatus.RETRY_PENDING.getCode());
        message.setNextRetryTime(calculateNextRetryTime());
        
        // 3. 可选：写入本地文件作为备份
        backupToLocalFile(message);
        
        // 4. 发送告警
        alertService.sendDegradationAlert(message);
    }
    
    /**
     * 人工干预接口
     */
    @ApiOperation("手动重试失败消息")
    public void manualRetry(List<Long> messageIds) {
        for (Long messageId : messageIds) {
            try {
                LocalMessage message = localMessageService.findById(messageId);
                messageSender.resend(message);
            } catch (Exception e) {
                log.error("手动重试消息失败: {}", messageId, e);
            }
        }
    }
}
```

## 9. 总结

### 9.1 方案优势
1. **可靠性高**：消息持久化，避免丢失
2. **性能较好**：异步处理，减少同步阻塞
3. **扩展性强**：易于增加新的消费者
4. **容错性好**：完善的补偿和重试机制

### 9.2 适用场景
- 异步通知类业务
- 对实时性要求不高的场景
- 需要保证最终一致性的分布式事务
- 跨系统数据同步

### 9.3 注意事项
1. 消息表需要与业务数据库在同一事务中
2. 需要处理好幂等性问题
3. 补偿频率需要根据业务特点调整
4. 注意监控消息积压情况
