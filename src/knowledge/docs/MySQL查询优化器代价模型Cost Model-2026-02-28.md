# MySQL查询优化器代价模型 (Cost Model) 技术文档

## 1. 概述

MySQL查询优化器的核心任务是为一条SQL语句找到一个**执行效率最高**的查询计划。为了实现这一目标，优化器需要对所有可能执行计划的“代价”进行评估和比较，并选择代价最低的那个。

**代价模型** 就是优化器内部用于量化计算不同执行计划资源消耗的数学模型。它将查询执行过程中的各项操作（如磁盘I/O、CPU计算、内存使用等）抽象为可量化的“代价”单位，从而为优化器的决策提供客观依据。

## 2. 核心思想

代价模型的核心思想是：**执行计划的代价 = 所有相关操作代价的总和**。

优化器在生成执行计划树时，会自底向上地计算每个操作符的代价，最终得到根节点（即整个查询）的总代价。这个总代价主要用于：
*   **选择表的访问方式**：全表扫描 vs. 索引扫描 vs. 索引范围扫描。
*   **选择多表连接顺序**：在N个表连接时，评估不同连接顺序的代价。
*   **选择连接算法**：使用嵌套循环连接、哈希连接还是排序合并连接。
*   **选择子查询的执行策略**：物化、半连接转换等。

## 3. MySQL代价模型的主要构成参数

MySQL的代价计算基于一组可配置的“代价常量”，这些常量存储在`mysql`系统库的`server_cost`和`engine_cost`表中（MySQL 5.7及以上版本）。它们代表了不同类型操作的基准开销。

### 3.1 `server_cost`表 (服务器层操作代价)
| cost_name | 默认值 | 描述 |
| :--- | :--- | :--- |
| `row_evaluate_cost` | 0.1 | 评估一条记录是否符合条件的CPU代价。如果查询需要检查大量行，此代价会显著增加。 |
| `key_compare_cost` | 0.1 | 比较索引键值或进行排序操作的CPU代价。 |
| `memory_temptable_create_cost` | 2.0 | 创建基于内存的临时表的代价。 |
| `memory_temptable_row_cost` | 0.2 | 向内存临时表写入或读取一条记录的代价。 |
| `disk_temptable_create_cost` | 40.0 | 创建基于磁盘的临时表的代价（当内存临时表超过`tmp_table_size`时）。 |
| `disk_temptable_row_cost` | 1.0 | 向磁盘临时表写入或读取一条记录的代价。 |

### 3.2 `engine_cost`表 (存储引擎层I/O代价)
| cost_name | 默认值 | 描述 |
| :--- | :--- | :--- |
| `io_block_read_cost` | 1.0 | 从磁盘或缓冲池中读取一个数据块（页）的代价。这是影响是否使用索引的关键参数。 |
| `memory_block_read_cost` | 0.25 | 从内存（缓冲池）中读取一个数据块（页）的代价。 |

**注意**：`engine_cost`表还可以为不同的存储引擎（如InnoDB、MyISAM）配置不同的代价。

## 4. 代价计算示例

### 4.1 单表查询代价估算

假设有一个表 `t`， 包含 10，000 条记录，每页可存放 100 条记录，即共有约 100 个数据页。

**场景A：全表扫描**
*   I/O代价：假设所有页都在内存中（缓冲池命中率高），则代价为 `100页 * memory_block_read_cost (0.25) = 25`。
*   CPU代价：需要检查所有行，代价为 `10000行 * row_evaluate_cost (0.1) = 1000`。
*   **总代价** ≈ `25 + 1000 = 1025`。

**场景B：使用索引范围扫描**
*   假设索引高度为3，需要读取3个索引页。
*   通过索引筛选后，预计需要回表访问 100 条主键记录（假设这100条记录分布在 20 个不同的数据页中）。
*   I/O代价：`(3 + 20)页 * 0.25 = 5.75`。
*   CPU代价：检查索引键和回表后筛选，代价约为 `100行 * 0.1 * 某个系数`。
*   **总代价** ≈ `5.75 + 10 = 15.75`。

**对比**：优化器会计算`1025`与`15.75`，显然选择索引扫描的计划。

### 4.2 连接查询代价估算

对于嵌套循环连接（Nested-Loop Join），其代价估算公式可简化为：
`连接代价 = 访问驱动表的代价 + (驱动表扇出记录数 * 访问被驱动表的代价)`

假设驱动表A筛选后有100条记录，访问A的代价为10。对于A中的每一条记录，都需要去表B进行一次索引查找，每次查找的代价为1。
*   连接总代价 = `10 + (100 * 1) = 110`。

优化器会估算不同表作为驱动表时的连接代价，以及不同连接顺序（A JOIN B vs. B JOIN A）的总代价，最终选择最低的方案。

## 5. 代价模型的实际应用与调整

### 5.1 查看查询代价
使用 `EXPLAIN FORMAT=JSON` 可以输出详细的代价信息。
```sql
EXPLAIN FORMAT=JSON SELECT * FROM t WHERE id = 1;
```
在输出的JSON中，可以找到 `query_cost` 字段，它就是优化器估算的该执行计划的总代价。

### 5.2 调整代价常量
如果优化器因代价估算不准而持续选择非最优计划（例如，高估索引代价或低估全表扫描代价），可以尝试调整代价常量。

**步骤**：
1. 查看当前配置：
   ```sql
   SELECT * FROM mysql.server_cost;
   SELECT * FROM mysql.engine_cost;
   ```

2. 更新配置（例如，认为磁盘I/O比默认更昂贵）：
   ```sql
   UPDATE mysql.engine_cost SET cost_value = 2.0
     WHERE cost_name = ‘io_block_read_cost‘;
   FLUSH OPTIMIZER_COSTS;
   ```
   **警告**：此操作需要数据库管理员权限，且调整不当可能导致更差的性能。务必在测试环境验证，并理解其影响。

### 5.3 代价模型的局限性与统计信息
代价模型的准确性高度依赖于**表统计信息**。如果统计信息过期（如行数、索引基数不准确），代价估算将严重偏离实际情况，导致优化器做出错误选择。

关键统计信息包括：
*   `TABLE_ROWS`：表中估计的行数。
*   `CARDINALITY`：索引列中不同值的数量估计。
*   数据分布直方图（MySQL 8.0引入）：提供了列值分布的非均匀性信息，对于 `WHERE col > 100` 这类查询的估算至关重要。

**建议**：定期执行 `ANALYZE TABLE` 来更新表的统计信息。

## 6. 总结

MySQL的代价模型是一个将查询执行过程量化为成本数字的复杂内部机制。通过理解其核心参数和估算逻辑，数据库管理员和开发者可以：
1.  **解读** `EXPLAIN` 的输出，理解优化器选择特定计划的原因。
2.  **诊断** 性能问题，识别是否因代价估算错误导致了次优计划。
3.  **干预** 优化器行为，在极少数情况下通过调整代价常量或更新统计信息来引导其做出更优决策。

然而，在大多数生产环境中，保持准确的统计信息并信任优化器的选择是最佳策略。代价模型的调整应被视为一种高级的、有风险的优化手段，需谨慎使用。
