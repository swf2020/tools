## Kafka消费幂等性技术方案：本地去重表与业务唯一键

---

### 1. 引言

在分布式消息系统中，消费端的**消息重复消费**是常见问题。Kafka自身提供的“至少一次”（at-least-once）语义，可能在以下场景导致重复：
- 消费者提交偏移量后，在业务处理完成前宕机（自动提交偏移量）
- 消费者重启后从最近提交的偏移量重新消费
- 分区重平衡期间

**消费幂等性**指：同一消息被消费多次，其业务效果与消费一次相同。本文探讨两种实现方案：**本地去重表**与**业务唯一键**。

---

### 2. 本地去重表方案

#### 2.1 原理
在消费者本地维护一个已处理消息的缓存，通过唯一标识判断消息是否已被消费。

#### 2.2 关键设计

**去重键生成**：
```java
String dedupeKey = topic + "-" + partition + "-" + offset; // 方案1：精确但无法跨分区迁移
String dedupeKey = message.getMsgId(); // 方案2：要求生产者生成全局唯一ID
```

**存储方案对比**：
| 存储类型 | 优点 | 缺点 | 适用场景 |
|---------|------|------|---------|
| 内存Map（LRU） | 性能极高（O(1)） | 重启丢失、内存限制 | 数据量小、允许少量重复 |
| RocksDB/LevelDB | 持久化、高性能 | 需磁盘空间、配置复杂 | 大数据量、允许重启 |
| Redis集群 | 分布式、可共享 | 网络开销、依赖外部服务 | 多消费者实例共享去重 |

#### 2.3 实现示例
```java
public class LocalDeduplicationConsumer {
    private final Cache<String, Boolean> processedCache;
    
    public void consume(ConsumerRecord<String, String> record) {
        String dedupeKey = buildDedupeKey(record);
        
        if (processedCache.getIfPresent(dedupeKey) != null) {
            log.warn("重复消息，跳过处理: {}", dedupeKey);
            return; // 幂等返回
        }
        
        // 业务处理
        processBusiness(record.value());
        
        // 记录已处理
        processedCache.put(dedupeKey, true);
        
        // 异步提交偏移量
        commitOffsetAsync(record);
    }
    
    private String buildDedupeKey(ConsumerRecord<String, String> record) {
        return String.format("%s-%d-%d", 
            record.topic(), 
            record.partition(), 
            record.offset());
    }
}
```

#### 2.4 存储优化策略
```yaml
# 配置示例：Guava Cache
local-dedupe:
  cache:
    max-size: 100000
    expire-after-write: 2h  # 基于时间窗口去重
    cleanup-policy: lru
```

#### 2.5 注意事项
- **时间窗口**：设置合理的过期时间，避免存储无限增长
- **内存管理**：监控缓存命中率，调整LRU策略
- **分区迁移**：Consumer重平衡会导致本地缓存失效，需结合业务唯一键

### 3. 业务唯一键方案

#### 3.1 原理
利用业务本身的唯一标识（如订单ID、交易流水号）实现幂等，将去重逻辑下沉到业务层。

#### 3.2 实现模式

**模式1：数据库唯一约束**
```sql
CREATE TABLE message_processed (
    id BIGINT AUTO_INCREMENT,
    business_key VARCHAR(64) NOT NULL,
    msg_timestamp TIMESTAMP,
    UNIQUE INDEX uk_business_key (business_key)
);
```

**模式2：乐观锁版本控制**
```java
public void processOrder(OrderMessage msg) {
    int updated = orderDao.updateOrderStatus(
        msg.getOrderId(), 
        msg.getVersion(), // 消息携带版本号
        OrderStatus.PROCESSED
    );
    
    if (updated == 0) {
        // 版本冲突或已处理，幂等返回
        log.info("订单已处理: {}", msg.getOrderId());
    }
}
```

**模式3：状态机幂等**
```java
public class OrderProcessor {
    public void process(String orderId, OrderEvent event) {
        Order order = orderDao.selectById(orderId);
        
        // 状态机校验
        if (!order.canAccept(event)) {
            throw new IllegalStateException("订单当前状态不可处理此事件");
        }
        
        // 状态转换
        order.applyEvent(event);
        orderDao.update(order);
    }
}
```

#### 3.3 完整示例
```java
@Component
public class BusinessDeduplicationService {
    
    @Transactional
    public void processWithDeduplication(OrderMessage message) {
        // 1. 插入去重记录（数据库唯一约束保证）
        try {
            dedupeDao.insertProcessRecord(
                message.getOrderId(), 
                message.getMsgId()
            );
        } catch (DuplicateKeyException e) {
            log.warn("订单已处理: {}", message.getOrderId());
            return; // 幂等返回
        }
        
        // 2. 业务处理（此时保证唯一性）
        orderService.processOrder(message);
        
        // 3. 可选：记录处理成功
        dedupeDao.markProcessSuccess(message.getOrderId());
    }
}
```

### 4. 方案对比与选型

| 维度 | 本地去重表 | 业务唯一键 |
|------|-----------|------------|
| **精确性** | 依赖offset，分区迁移会失效 | 业务级精确去重 |
| **存储开销** | 消费者内存/磁盘 | 业务数据库 |
| **性能影响** | 微秒级判断 | 数据库IO开销 |
| **分布式支持** | 单实例有效 | 天然支持分布式 |
| **容灾恢复** | 缓存丢失导致重复 | 持久化可靠 |
| **业务耦合** | 无业务侵入 | 需业务字段配合 |
| **适用场景** | 日志处理、监控数据 | 订单、交易等核心业务 |

### 5. 混合方案实践

#### 5.1 两级去重架构
```java
public class HybridDeduplicationProcessor {
    // 第一级：本地缓存快速过滤
    private final Cache<String, Boolean> localCache;
    
    // 第二级：数据库持久化去重
    private final DeduplicationService dedupeService;
    
    public void process(ConsumerRecord<String, String> record) {
        String businessKey = extractBusinessKey(record);
        
        // 1. 本地缓存检查
        if (localCache.getIfPresent(businessKey) != null) {
            return;
        }
        
        // 2. 数据库唯一性检查
        if (!dedupeService.tryAcquireLock(businessKey)) {
            localCache.put(businessKey, true); // 预热本地缓存
            return;
        }
        
        try {
            // 3. 业务处理
            processBusiness(record.value());
            
            // 4. 更新状态
            dedupeService.markProcessed(businessKey);
            localCache.put(businessKey, true);
            
        } finally {
            dedupeService.releaseLock(businessKey);
        }
    }
}
```

#### 5.2 实时与离线协同
```
实时链路：
消息 → 本地缓存 → 业务处理 → 异步提交

补偿链路：
定时任务扫描 → 疑似重复数据 → 业务校验 → 人工干预
```

### 6. 生产环境建议

#### 6.1 监控指标
- 重复消息率：`duplicate_messages_total`
- 去重缓存命中率：`dedupe_cache_hit_ratio`
- 业务唯一键冲突数：`business_key_conflicts`

#### 6.2 配置示例
```yaml
kafka:
  consumer:
    enable-auto-commit: false
    auto-offset-reset: latest
    
deduplication:
  strategy: hybrid
  local-cache:
    size: 50000
    ttl-minutes: 120
  business-key:
    enabled: true
    source-field: orderId
  fallback:
    enable-dlq: true
    max-retries: 3
```

#### 6.3 灾备策略
1. **数据恢复后**：重置消费者偏移量，依赖业务幂等性保证正确性
2. **缓存丢失时**：允许短期重复，由业务层兜底
3. **数据库不可用**：降级为本地缓存模式，记录异常日志

---

### 7. 总结

1. **轻量级场景**：优先使用本地去重表，注意设置合理过期时间
2. **核心业务**：必须使用业务唯一键，结合数据库约束
3. **高并发场景**：采用混合方案，本地缓存+数据库校验
4. **最终一致性**：幂等性是基础，需结合重试、死信队列等机制

通过合理选择去重策略，可在保证消息可靠性的同时，避免重复消费带来的业务问题。
