# TCP三次握手状态机技术文档

## 1. 文档概述

### 1.1 文档目的
本文档详细描述TCP（传输控制协议）连接建立过程中的三次握手状态机，涵盖客户端和服务端在各阶段的状态转换、报文交互机制以及异常处理流程。

### 1.2 目标读者
- 网络协议开发工程师
- 网络运维工程师
- 计算机网络学习者
- 系统架构师

### 1.3 技术背景
TCP是一种面向连接的、可靠的传输层协议，在传输数据前必须建立端到端的连接。三次握手是TCP连接建立的标准化过程，确保双方能够同步初始序列号并协商连接参数。

## 2. TCP三次握手核心概念

### 2.1 三次握手的基本流程
```
客户端                              服务端
  |                                 |
  |        SYN (seq=x)              |
  |------------------------------->|
  |                                 | 状态: LISTEN → SYN_RCVD
  |                                 |
  |    SYN-ACK (seq=y, ack=x+1)     |
  |<-------------------------------|
  | 状态: SYN_SENT → ESTABLISHED    |
  |                                 |
  |        ACK (ack=y+1)            |
  |------------------------------->|
  |                                 | 状态: SYN_RCVD → ESTABLISHED
```

### 2.2 关键字段说明
- **SYN**: 同步序列号标志，置1表示请求建立连接
- **ACK**: 确认标志，置1表示确认号有效
- **seq**: 序列号（32位），标识报文段中第一个字节的序号
- **ack**: 确认号（32位），期望收到的下一个字节的序号

## 3. 状态机详细解析

### 3.1 服务端状态机

#### 3.1.1 LISTEN状态
```
状态: LISTEN
条件: 服务端调用listen()后等待连接
行为:
  - 维护未完成连接队列（SYN队列）
  - 接收SYN报文
  - 验证端口和IP地址
超时处理: 无主动超时
转换条件: 收到合法SYN → SYN_RCVD
```

#### 3.1.2 SYN_RCVD状态
```
状态: SYN_RCVD
条件: 收到SYN报文，已发送SYN-ACK
行为:
  - 分配连接资源
  - 维护半连接信息
  - 启动重传计时器
超时处理: SYN-ACK重传（指数退避）
转换条件:
  - 收到ACK → ESTABLISHED
  - 超时或RST → CLOSED
```

#### 3.1.3 ESTABLISHED状态
```
状态: ESTABLISHED
条件: 完成三次握手
行为: 正常数据传输
```

### 3.2 客户端状态机

#### 3.2.1 CLOSED状态
```
状态: CLOSED
条件: 初始状态或连接关闭
行为: 等待应用层调用connect()
```

#### 3.2.2 SYN_SENT状态
```
状态: SYN_SENT
条件: 发送SYN后等待SYN-ACK
行为:
  - 启动连接超时计时器
  - 维护连接参数
超时处理: SYN重传（最大重试次数限制）
转换条件:
  - 收到SYN-ACK → ESTABLISHED
  - 收到RST → CLOSED
  - 超时 → CLOSED
```

#### 3.2.3 ESTABLISHED状态
```
状态: ESTABLISHED
条件: 收到SYN-ACK并发送ACK
行为: 正常数据传输
```

## 4. 状态转换矩阵

### 4.1 客户端状态转换表
| 当前状态 | 事件/条件 | 动作 | 下一状态 |
|---------|----------|------|---------|
| CLOSED | connect()调用 | 发送SYN，启动计时器 | SYN_SENT |
| SYN_SENT | 收到SYN-ACK | 发送ACK | ESTABLISHED |
| SYN_SENT | 收到RST | 释放资源 | CLOSED |
| SYN_SENT | 超时重传超过限制 | 报告错误 | CLOSED |

### 4.2 服务端状态转换表
| 当前状态 | 事件/条件 | 动作 | 下一状态 |
|---------|----------|------|---------|
| LISTEN | 收到SYN | 发送SYN-ACK，启动计时器 | SYN_RCVD |
| SYN_RCVD | 收到ACK | 将连接移至完成队列 | ESTABLISHED |
| SYN_RCVD | 收到RST | 释放资源 | LISTEN |
| SYN_RCVD | 超时未收到ACK | 释放资源 | LISTEN |

## 5. 数据包交互细节

### 5.1 第一次握手（SYN）
```
客户端构造SYN报文:
  - 设置SYN=1, ACK=0
  - 生成初始序列号ISN_c
  - seq = ISN_c
  - 可选: 包含MSS、窗口缩放等选项
```

### 5.2 第二次握手（SYN-ACK）
```
服务端回应SYN-ACK:
  - 设置SYN=1, ACK=1
  - 生成初始序列号ISN_s
  - seq = ISN_s
  - ack = ISN_c + 1
  - 携带协商参数
```

### 5.3 第三次握手（ACK）
```
客户端发送ACK:
  - 设置SYN=0, ACK=1
  - seq = ISN_c + 1
  - ack = ISN_s + 1
  - 可携带第一个数据包（捎带确认）
```

## 6. 异常处理机制

### 6.1 连接超时与重传
- **初始超时**: 通常为3秒
- **重传策略**: 指数退避算法
- **最大重试次数**: 通常为5-6次
- **总超时时间**: 约75-127秒

### 6.2 半连接队列管理
```
半连接队列（SYN队列）参数:
  - 最大长度: net.ipv4.tcp_max_syn_backlog
  - 超时时间: net.ipv4.tcp_synack_retries控制
保护机制: SYN Cookie（防止SYN Flood攻击）
```

### 6.3 RST处理
触发RST的条件：
1. 到不存在的端口的连接请求
2. 异常终止连接
3. 检测到半开连接

## 7. 性能优化与安全考虑

### 7.1 性能优化
- **TCP Fast Open**: 减少握手延迟
- **SYN缓存**: 重用握手参数
- **时间戳选项**: 防止序列号回绕

### 7.2 安全防护
- **SYN Cookie机制**: 防御SYN Flood攻击
- **SYN代理**: 验证客户端真实性
- **连接限制**: 限制单个IP的连接数

```bash
# Linux 系统

ss -t state listening
cat /proc/net/tcp | grep -E "0A|01|03"

# 查看半连接队列溢出
netstat -s | grep -i "listen"
```

1. **连接建立失败**
   - 检查防火墙规则
   - 验证端口监听状态
   - 检查半连接队列设置

2. **连接超时**
   - 网络延迟检测
   - 重传参数优化
   - 服务端负载检查
