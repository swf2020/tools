# TiDB 执行计划优化：统计信息与直方图机制详解

## 1. 摘要

本文档深入探讨 TiDB 执行计划优化中的核心组件——统计信息与直方图机制。通过详细分析统计信息的收集、维护及应用原理，帮助数据库管理员和开发者理解 TiDB 如何利用直方图优化查询性能，并提供实践指导以提升系统整体效率。

## 2. 引言

### 2.1 执行计划优化的重要性
TiDB 作为分布式数据库，查询性能高度依赖于执行计划的优劣。执行计划决定了：
- 数据访问路径（索引扫描 vs 全表扫描）
- 多表关联策略（HashJoin、IndexJoin 等）
- 聚合计算的下推与分布
- 子查询的处理方式

### 2.2 统计信息的关键作用
准确的统计信息是优化器生成高效执行计划的基石，主要包括：
- 表级别的行数、数据大小
- 列级别的基数（cardinality）、NULL 值比例
- 数据分布特征（直方图、CMSketch）
- 索引信息（唯一值数量、索引长度）

## 3. 统计信息体系架构

### 3.1 统计信息组成

## 4. 直方图机制深度解析

### 4.1 直方图基础概念

#### 4.1.1 直方图的作用
直方图通过将数据分布划分为多个桶（bucket），帮助优化器理解：
- 数据的实际分布形态（均匀、偏斜、多峰）
- 查询条件的选择性估算
- 范围查询的代价估算

#### 4.1.2 桶（Bucket）结构
```go
type Bucket struct {
    Count      int64   // 桶内数据行数
    LowerBound []byte  // 下界值（含）
    UpperBound []byte  // 上界值（含）
    Repeats    int64   // 上界值的重复次数
    NDV        int64   // 桶内不同值的数量（NDV）
}
```

### 4.2 直方图类型

#### 4.2.1 等高直方图（Equi-height Histogram）
- 每个桶包含大致相同数量的行
- 适用于均匀分布的数据
- 桶边界自动确定

#### 4.2.2 等宽直方图（Equi-width Histogram）
- 每个桶覆盖相同的值域范围
- 适用于特定业务场景
- TiDB 默认使用等高直方图

### 4.3 直方图构建过程

#### 4.3.1 采样阶段
```sql
-- TiDB 使用 Reservoir Sampling 算法采样
-- 采样大小由 tidb_analyze_version 和 tidb_stats_sampling_rate 控制
SET GLOBAL tidb_stats_sampling_rate = 0.1;  -- 10% 采样率
```

#### 4.3.2 桶划分算法
```python
def build_histogram(data_samples, num_buckets):
    """
    构建直方图的核心算法
    """
    # 1. 对样本数据排序
    sorted_data = sort(data_samples)
    
    # 2. 计算每个桶的目标行数
    rows_per_bucket = len(sorted_data) / num_buckets
    
    # 3. 动态划分桶边界
    buckets = []
    current_bucket = Bucket()
    
    for value in sorted_data:
        current_bucket.add(value)
        
        if current_bucket.count >= rows_per_bucket:
            # 寻找合适的切分点
            if value != next_value:
                buckets.append(current_bucket)
                current_bucket = Bucket()
    
    return buckets
```

```sql

-- 控制统计信息收集版本
SET GLOBAL tidb_analyze_version = 2;  -- V2 版本支持更精确的统计
```

### 4.4 直方图的选择性估算

#### 4.4.1 等值查询估算
```python
def estimate_eq_selectivity(histogram, value):
    """
    估算等值查询的选择性
    """
    # 1. 查找值所在的桶
    bucket = find_bucket(histogram, value)
    
    if bucket is None:
        return 0.0
    
    # 2. 如果值等于桶边界
    if value == bucket.upper_bound:
        # 考虑重复值
        selectivity = bucket.repeats / bucket.count
    else:
        # 假设桶内均匀分布
        selectivity = (1 / bucket.ndv) * (bucket.count / total_rows)
    
    return selectivity
```

#### 4.4.2 范围查询估算
```python
def estimate_range_selectivity(histogram, lower, upper):
    """
    估算范围查询的选择性
    """
    total_selectivity = 0.0
    
    # 1. 查找范围内的所有桶
    affected_buckets = find_affected_buckets(histogram, lower, upper)
    
    for bucket in affected_buckets:
        # 2. 计算桶内满足条件的比例
        bucket_ratio = calculate_bucket_ratio(bucket, lower, upper)
        
        # 3. 乘以桶的选择性
        bucket_selectivity = bucket_ratio * (bucket.count / total_rows)
        total_selectivity += bucket_selectivity
    
    return total_selectivity
```

## 5. 直方图在执行计划优化中的应用

### 5.1 选择率（Selectivity）估算

#### 5.1.1 单列条件估算
```sql
-- 示例查询
SELECT * FROM orders WHERE order_amount > 1000;

-- 优化器使用直方图估算
-- 1. 定位 order_amount > 1000 影响的桶
-- 2. 计算每个桶内满足条件的比例
-- 3. 累加得到总选择率
```

#### 5.1.2 多列条件估算
TiDB 使用以下策略处理多列条件：
```sql
-- 独立假设（默认）
-- selectivity(col1 AND col2) = selectivity(col1) * selectivity(col2)

-- 使用扩展统计信息（TiDB 5.0+）
ALTER TABLE table_name ADD STATS_EXTENDED type_name (column_list);
ANALYZE TABLE table_name;
```

### 5.2 连接（Join）顺序优化

#### 5.2.1 基于代价的连接排序
```python
def optimize_join_order(tables, predicates):
    """
    基于统计信息的连接顺序优化
    """
    # 1. 估算每个表的大小和选择率
    table_stats = estimate_table_statistics(tables, predicates)
    
    # 2. 使用动态规划寻找最优连接顺序
    best_plan = dynamic_programming_join_ordering(table_stats)
    
    return best_plan
```

#### 5.2.2 连接选择性估算
```sql
-- 示例：两个表的等值连接
SELECT * 
FROM orders o 
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.status = 'completed';

-- 优化器考虑：
-- 1. orders 表中 status='completed' 的选择率
-- 2. customers 表的基数
-- 3. 连接键的分布特征
```

### 5.3 索引选择优化

#### 5.3.1 索引代价模型
TiDB 为每个候选索引计算代价：
```
索引代价 = IO代价 + CPU代价

IO代价 = 索引扫描行数 * io_cost_per_row
CPU代价 = 数据处理行数 * cpu_cost_per_row
```

#### 5.3.2 使用直方图改进索引选择
```sql
-- 示例：范围查询的索引选择
SELECT * FROM orders 
WHERE order_date BETWEEN '2023-01-01' AND '2023-06-30'
AND total_amount > 1000;

-- 优化器使用直方图：
-- 1. 估算 order_date 条件的选择率
-- 2. 估算 total_amount 条件的选择率
-- 3. 计算各索引的扫描代价
-- 4. 选择代价最低的索引
```

## 8. 常见问题与解决方案

### 8.1 统计信息不准确问题

#### 8.1.1 症状识别
- 执行计划频繁变化
- 查询性能不稳定
- EXPLAIN 显示估算行数与实际差异大

#### 8.1.2 解决方案
```sql
-- 1. 强制重新收集统计信息
ANALYZE TABLE table_name WITH BUCKETS 512;

-- 2. 调整采样率
SET GLOBAL tidb_stats_sampling_rate = 0.3;
ANALYZE TABLE table_name;

-- 3. 手动更新特定统计信息
UPDATE mysql.stats_histograms 
SET distinct_count = {correct_value}
WHERE table_id = {table_id} AND column_id = {column_id};
```

### 8.2 直方图桶数量优化

#### 8.2.1 桶数量选择策略
```
数据分布特征             推荐桶数
-------------------     ----------
高度均匀                 64-128
轻度偏斜                 128-256
严重偏斜                 256-512
极值较多                 512-1024
```

#### 8.2.2 动态调整示例
```sql
-- 根据数据特征调整桶数
SET @bucket_count = (
    SELECT CASE
        WHEN COUNT(DISTINCT column_name) < 100 THEN 64
        WHEN COUNT(DISTINCT column_name) < 10000 THEN 256
        ELSE 512
    END
    FROM table_name
);

SET @analyze_sql = CONCAT(
    'ANALYZE TABLE table_name WITH BUCKETS ', 
    @bucket_count
);

PREPARE stmt FROM @analyze_sql;
EXECUTE stmt;
```
