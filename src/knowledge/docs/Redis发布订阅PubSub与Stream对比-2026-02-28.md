# Redis发布订阅(Pub/Sub)与Stream对比技术文档

## 1. 概述

Redis提供了两种不同的消息传递机制：传统的发布订阅(Pub/Sub)模式和Redis 5.0引入的Stream类型。两者都可用于实现消息系统，但在设计理念、功能特性和适用场景上存在显著差异。

## 2. Redis发布订阅(Pub/Sub)

### 2.1 核心特性

**无状态消息传递**
- 消息发送后即消失，不存储历史记录
- 订阅者只能接收订阅后发布的消息
- 无法重新消费已发送的消息

**即时消费模型**
- 消息实时推送给所有在线订阅者
- 离线订阅者会丢失未连接期间的消息
- 基于通道的广播机制

**基本命令**
```redis
# 发布消息
PUBLISH channel "message"

# 订阅通道
SUBSCRIBE channel1 channel2

# 模式订阅
PSUBSCRIBE news.*
```

### 2.2 架构特点
```
发布者 ──发布消息──> Redis服务器 ──推送消息──> 订阅者1
                                  └─推送消息──> 订阅者2
                                  └─推送消息──> 订阅者3
```

## 3. Redis Stream

### 3.1 核心特性

**持久化消息存储**
- 消息以追加方式存储在内存中
- 可配置最大长度或基于时间的保留策略
- 支持历史消息重放

**消费者组模式**
- 多个消费者组独立消费同一Stream
- 组内消费者负载均衡
- 支持消息确认机制

**精确消费控制**
- 支持基于ID的消费位置管理
- 提供未确认消息待处理列表
- 可查询消费者状态

### 3.2 基本命令
```redis
# 添加消息到Stream
XADD mystream * field1 value1 field2 value2

# 读取消息
XREAD COUNT 2 STREAMS mystream 0

# 创建消费者组
XGROUP CREATE mystream mygroup 0

# 消费者组消费
XREADGROUP GROUP mygroup consumer1 COUNT 1 STREAMS mystream >
```

## 4. 功能对比

### 4.1 特性对比表

| 特性维度 | Pub/Sub | Stream |
|---------|---------|---------|
| **消息持久化** | 不支持，消息即发即失 | 支持，可配置保留策略 |
| **历史消息** | 无法获取订阅前的消息 | 支持历史消息查询和重放 |
| **消费者模型** | 广播模式，所有订阅者接收相同消息 | 支持消费者组，支持负载均衡 |
| **消息确认** | 无确认机制 | 支持ACK确认，防止消息丢失 |
| **离线处理** | 离线期间消息丢失 | 离线后可重新消费未处理消息 |
| **性能影响** | 极低，无状态操作 | 略高，需要维护消费状态 |
| **内存使用** | 基本不占用额外内存 | 根据保留策略占用内存 |
| **适用场景** | 实时通知、广播消息 | 消息队列、事件溯源、日志收集 |

### 4.2 数据模型对比

**Pub/Sub消息流**
```
时间轴: t1────t2────t3────t4────t5
发布者:  A────B────C────D────E
订阅者:  A────B────C────D────E (在线时)
         └───缺失──┘ (离线时)
```

**Stream消息流**
```
Stream: [A][B][C][D][E][F][G]...
         ↑      ↑    ↑
         消费者组1   消费者组2
         消费者1     消费者1
         消费者2
```

## 5. 使用场景分析

### 5.1 适合使用Pub/Sub的场景

1. **实时通知系统**
   - 在线用户状态更新
   - 实时聊天消息推送
   - 即时数据更新广播

2. **事件广播**
   - 配置变更通知
   - 系统状态广播
   - 轻量级事件分发

3. **临时消息传递**
   - 无需持久化的实时通信
   - 客户端实时指令推送

### 5.2 适合使用Stream的场景

1. **消息队列应用**
   - 任务队列处理
   - 订单处理流程
   - 异步任务分发

2. **事件溯源**
   - 用户行为日志
   - 系统操作审计
   - 状态变更历史

3. **数据管道**
   - 日志收集与分发
   - 数据ETL处理
   - 微服务间通信

## 6. 性能与资源考量

### 6.1 性能特征
- **Pub/Sub**: O(N+M)，N为通道数，M为订阅者数，性能极高
- **Stream**: O(log N)大部分操作，需要维护更多状态

### 6.2 内存使用
- **Pub/Sub**: 几乎不占用额外内存
- **Stream**: 需要存储消息内容，可配置最大内存使用

### 6.3 网络开销
- **Pub/Sub**: 连接保持期间持续推送
- **Stream**: 按需拉取，可控制消费速率

## 7. 选型建议

### 7.1 选择Pub/Sub的条件
- ✅ 消息无需持久化
- ✅ 允许消息丢失
- ✅ 广播模式满足需求
- ✅ 对实时性要求极高
- ✅ 消费者始终在线或允许丢失消息

### 7.2 选择Stream的条件
- ✅ 需要消息持久化和重放
- ✅ 需要消费者组和负载均衡
- ✅ 需要消息确认机制
- ✅ 处理离线消费者场景
- ✅ 实现标准消息队列模式

## 8. 混合使用模式

在某些复杂场景中，可以结合使用两种模式：

```redis
# 使用Stream作为可靠消息队列
XADD orders * user_id 123 item "book" quantity 2

# 使用Pub/Sub发送实时通知
PUBLISH notifications "New order received"

# 消费者组处理订单
XREADGROUP GROUP order_processors consumer1 STREAMS orders >
```

## 9. 限制与注意事项

### 9.1 Pub/Sub限制
- 无持久化，重启后所有订阅关系丢失
- 大量订阅者时网络连接管理复杂
- 不支持消息回溯

### 9.2 Stream限制
- 长期运行可能占用较多内存
- 消费者组管理相对复杂
- Redis集群模式下跨节点消费组有限制

## 10. 总结

Redis Pub/Sub和Stream代表了两种不同的消息传递哲学：
- **Pub/Sub**是轻量级、实时、无状态的发布-订阅系统
- **Stream**是功能丰富、可靠、有状态的消息流系统

选择哪种方案取决于具体业务需求：
- 对于简单的实时通知和广播，Pub/Sub是更轻量的选择
- 对于需要可靠性、持久化和复杂消费者模式的消息处理，Stream是更合适的选择

在实际应用中，两者可以互补使用，构建更加健壮的消息处理架构。
