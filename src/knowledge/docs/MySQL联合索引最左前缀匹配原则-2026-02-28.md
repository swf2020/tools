# MySQL联合索引最左前缀匹配原则技术文档

## 1. 概述

联合索引（Composite Index）也称为复合索引或多列索引，是MySQL中对多个列同时建立的索引。最左前缀匹配原则（Leftmost Prefix Principle）是联合索引的核心使用原则，它规定了在使用联合索引进行查询时，索引列的使用顺序必须从最左边开始，并且不能跳过中间的列。

## 2. 联合索引的结构

### 2.1 索引存储方式
MySQL使用B+树结构存储联合索引，索引键按照列定义的顺序进行排序：
- 首先按第一列排序
- 第一列相同的情况下按第二列排序
- 以此类推

**示例：** 索引 `idx_name_age_city(name, age, city)` 的存储顺序：
```
Alex, 20, Beijing
Alex, 25, Shanghai
Bob, 30, Beijing
Bob, 30, Shanghai
Bob, 35, Guangzhou
```

### 2.2 索引有效性的层级关系
联合索引的可用性呈现层级结构：
- 第一列 → 可以单独使用
- 第一列+第二列 → 可以组合使用
- 第一列+第二列+第三列 → 可以完整使用

## 3. 最左前缀匹配原则详解

### 3.1 基本原则
当查询条件包含联合索引的前N个列时（N≥1），索引可以被有效利用。查询条件必须**从最左边的列开始，且不能跳过中间的列**。

### 3.2 有效使用场景

#### 3.2.1 全列匹配
```sql
-- 索引：idx_a_b_c(a, b, c)
SELECT * FROM table WHERE a = 1 AND b = 2 AND c = 3;  -- √ 完全使用索引
```

#### 3.2.2 左前缀匹配
```sql
-- 索引：idx_a_b_c(a, b, c)
SELECT * FROM table WHERE a = 1 AND b = 2;  -- √ 使用前两列
SELECT * FROM table WHERE a = 1;            -- √ 使用第一列
```

#### 3.2.3 范围查询后的列
```sql
-- 索引：idx_a_b_c(a, b, c)
SELECT * FROM table WHERE a = 1 AND b > 2 AND c = 3;  
-- √ a使用等值查询，b使用范围查询，c在范围查询后无法使用索引
```

### 3.3 无效使用场景

#### 3.3.1 跳过左边列
```sql
-- 索引：idx_a_b_c(a, b, c)
SELECT * FROM table WHERE b = 2;                    -- × 缺少最左列a
SELECT * FROM table WHERE b = 2 AND c = 3;          -- × 缺少最左列a
SELECT * FROM table WHERE a = 1 AND c = 3;          -- × 跳过中间列b
```

#### 3.3.2 非连续使用
```sql
-- 索引：idx_a_b_c_d(a, b, c, d)
SELECT * FROM table WHERE a = 1 AND c = 3;          -- × 只能用到a，无法用到c
SELECT * FROM table WHERE a = 1 AND b > 2 AND d = 4; -- × 只能用到a,b，无法用到d
```

## 5. 特殊情况和优化技巧

### 5.1 范围查询的影响
范围查询（`>`, `<`, `BETWEEN`, `LIKE`等）会截断后续索引列的使用：
```sql
-- 索引：idx_a_b_c(a, b, c)
SELECT * FROM table WHERE a = 1 AND b > 2 AND c = 3;
-- 只能使用到a,b列，c列无法使用索引
```

### 5.2 LIKE查询的注意事项
```sql
-- 索引：idx_name(name)
SELECT * FROM table WHERE name LIKE 'John%';    -- √ 前缀匹配，可以使用索引
SELECT * FROM table WHERE name LIKE '%John';    -- × 后缀匹配，无法使用索引
SELECT * FROM table WHERE name LIKE '%John%';   -- × 前后通配，无法使用索引
```

### 5.3 索引列顺序优化策略
#### 5.3.1 选择原则
1. **高频查询条件**放在最左边
2. **区分度高**的列放在左边
3. **等值查询**的列放在范围查询列前面
4. **经常需要排序**的列考虑加入索引

#### 5.3.2 示例比较
```sql
-- 方案1：区分度低的列在前
CREATE INDEX idx_gender_age ON users(gender, age);
-- gender只有2-3种值，区分度低

-- 方案2：区分度高的列在前（推荐）
CREATE INDEX idx_age_gender ON users(age, gender);
-- age区分度高，能更快缩小查询范围
```

### 5.4 覆盖索引优化
当查询的列全部包含在索引中时，无需回表查询：
```sql
-- 索引：idx_a_b_c(a, b, c)
SELECT a, b FROM table WHERE a = 1 AND b = 2;  -- √ 覆盖索引
SELECT * FROM table WHERE a = 1 AND b = 2;      -- × 需要回表查询其他列
```

## 6. 实际应用案例

### 6.1 电商订单查询优化
```sql
-- 原始设计
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    status TINYINT,
    create_time DATETIME,
    amount DECIMAL(10,2)
);

-- 常见查询
SELECT * FROM orders 
WHERE user_id = 1001 
  AND status = 1 
  AND create_time > '2024-01-01'
ORDER BY create_time DESC;

-- 优化方案：创建联合索引
CREATE INDEX idx_user_status_time ON orders(user_id, status, create_time);
-- user_id(等值) → status(等值) → create_time(范围+排序)
```

### 6.2 多条件筛选优化
```sql
-- 产品筛选场景
SELECT * FROM products 
WHERE category_id = 5 
  AND price BETWEEN 100 AND 500 
  AND brand_id = 10
ORDER BY sales_volume DESC;

-- 索引设计
CREATE INDEX idx_category_brand_price ON products(category_id, brand_id, price);
-- category_id和brand_id作为等值条件，price作为范围条件
```

## 8. 常见误区与注意事项

### 8.1 误区澄清
1. **误区**：只要查询条件包含索引列就能使用索引
   **正解**：必须从最左列开始，不能跳过中间列

2. **误区**：联合索引列顺序可以任意调整
   **正解**：列顺序决定索引的有效性，需要根据查询模式设计

3. **误区**：索引越多查询越快
   **正解**：索引会增加写操作成本，需要权衡利弊

### 8.2 最佳实践建议
1. **分析查询模式**：根据实际查询需求设计索引
2. **使用EXPLAIN分析**：定期检查查询执行计划
3. **避免过度索引**：每个表建议不超过5个索引
4. **定期维护**：使用`ANALYZE TABLE`更新索引统计信息
5. **监控性能**：关注索引的 selectivity（选择性）

## 9. 总结

MySQL联合索引的最左前缀匹配原则是数据库索引优化的核心知识。正确理解和应用这一原则可以：
- 显著提高查询性能
- 减少全表扫描的发生
- 降低数据库服务器负载
- 优化存储空间使用

在实际应用中，需要结合具体的业务查询模式，合理设计联合索引的列顺序，并通过执行计划分析验证索引使用效果，从而达到最佳的查询性能优化。
