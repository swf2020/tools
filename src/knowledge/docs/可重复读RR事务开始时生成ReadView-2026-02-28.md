# **InnoDB 可重复读隔离级别中 ReadView 的生成时机与机制**

## **1. 文档概述**

本文档旨在深入解析 MySQL InnoDB 存储引擎在 **可重复读（Repeatable Read, RR）** 隔离级别下，**ReadView（读视图）** 的核心机制——**在事务开始时生成**。重点阐述其工作原理、如何保障“可重复读”特性，并通过对比、示例和原理图进行说明。

**目标读者：** 数据库开发人员、架构师、对数据库事务隔离机制有深入理解需求的技术人员。

---

## **2. 核心概念定义**

### **2.1 可重复读 (Repeatable Read, RR)**
- **定义：** 事务隔离级别的一种。它确保在同一个事务中，多次读取同一范围的数据会返回相同的结果集，即使其他事务在此期间修改并提交了这些数据。这解决了“不可重复读”问题。
- **InnoDB 实现：** 通过 **多版本并发控制（MVCC）** 和 **ReadView** 机制来实现。

### **2.2 读视图 (ReadView)**
- **定义：** ReadView 是 InnoDB 为实现 MVCC 而创建的一个**一致性快照**。它定义了在某一时刻，当前活跃（未提交）事务的集合。事务根据此 ReadView 来判断数据行的某个版本是否对其可见。
- **关键组成（快照信息）：**
    1.  `m_ids`: 生成 ReadView 时，系统中**活跃事务**（已开始但未提交或回滚）的事务 ID 列表。
    2.  `min_trx_id`: 活跃事务中最小的事务 ID。
    3.  `max_trx_id`: 生成 ReadView 时，系统应分配给下一个事务的 ID（即当前最大事务 ID + 1）。
    4.  `creator_trx_id`: 创建该 ReadView 的事务自身的 ID（对于只读事务，该值可能为0）。

## **3. RR 隔离级别下 ReadView 的生成时机**

### **3.1 核心规则**
**在可重复读（RR）隔离级别下，一个事务在其第一次执行查询（如 `SELECT`）操作时，会生成一个独立的 ReadView。这个 ReadView 将用于该事务后续所有的普通 `SELECT` 查询，直至事务结束。**

### **3.2 详细说明与流程**
1.  **事务启动：** 当执行 `BEGIN` 或 `START TRANSACTION`（或关闭自动提交后执行第一条语句）时，事务开始，并获取一个唯一的事务 ID。
2.  **首次查询触发创建：** 事务执行第一条 `SELECT` 语句（非 `SELECT ... FOR UPDATE` 等加锁读）时，InnoDB 会为该事务**瞬间生成一个 ReadView**，捕获此刻系统的活跃事务状态。
3.  **快照的固定化：** 这个生成的 ReadView 会被**固化**，并关联到该事务的整个生命周期。
4.  **后续查询的一致性：** 在该事务后续的任何时间点执行新的 `SELECT` 查询，都会**复用这个最初生成的 ReadView** 来判断数据的可见性，而非重新生成。

### **3.3 对比：与读已提交（RC）的区别**
为了加深理解，此处对比 **读已提交（Read Committed, RC）** 隔离级别：

| 特性 | 可重复读 (RR) | 读已提交 (RC) |
| :--- | :--- | :--- |
| **ReadView 生成时机** | **事务开始时（首次查询）**生成一次，全程复用。 | **每条 `SELECT` 语句执行时**都会重新生成一个新的 ReadView。 |
| **可见性判断依据** | 始终基于事务开始时的“快照”。 | 基于每条语句执行时的最新“快照”。 |
| **效果** | 多次读取同一数据，结果一致（可重复读）。 | 每次读取都能看到其他事务已提交的最新修改（不可重复读）。 |

## **4. 如何通过 ReadView 实现可重复读**

### **4.1 数据行版本的可见性判断规则**
InnoDB 每行数据都有隐藏的 `DB_TRX_ID` 字段，记录最后修改它的事务ID。当需要判断某行数据的一个历史版本对当前事务是否可见时，会使用其 ReadView，并遵循以下规则（设数据行版本的 `trx_id` 为 `X`）：

1.  如果 `X` **小于** `min_trx_id`，说明该版本在 ReadView 生成前已提交，**可见**。
2.  如果 `X` **大于等于** `max_trx_id`，说明该版本在 ReadView 生成后才开启，**不可见**。
3.  如果 `X` 在 `[min_trx_id, max_trx_id)` 区间内：
    - 若 `X` **在 `m_ids` 列表中**，说明该版本由生成 ReadView 时仍活跃的事务创建，**不可见**。
    - 若 `X` **不在 `m_ids` 列表中**，说明该版本在生成 ReadView 时已提交，**可见**。
4.  如果 `X` 等于 `creator_trx_id`，说明是该事务自己修改的，**可见**。

**RR 事务全程使用同一套规则（同一个 ReadView）进行判断，因此对同一数据的可见性结论始终不变。**

### **4.2 原理示意图**
```
时间线：
T1: Trx-100 BEGIN
T2: Trx-100 第一次 SELECT (生成ReadView: m_ids=[], min=100, max=101) -> 看到数据 Version-1
T3: Trx-101 BEGIN 并 UPDATE 该行数据，生成 Version-2 (trx_id=101)
T4: Trx-101 COMMIT
T5: Trx-100 第二次 SELECT -> **复用T2时的ReadView** 判断：
        - Version-2 (trx_id=101) >= max_trx_id(101)？ 是 -> 不可见
        - 沿 Undo Log 找到 Version-1 (trx_id=90) < min_trx_id(100)？ 是 -> 可见
    => 仍然读取到 Version-1，实现可重复读。
```

## **5. 重要注意事项**

1.  **对当前事务的修改总是可见的：** 即使使用 RR，事务也能看到**自己在本事务内**修改的数据。
2.  **写操作（UPDATE/DELETE）的读是当前读：** `UPDATE`、`DELETE`、`SELECT ... FOR UPDATE` 等操作执行时，会读取数据行的**最新已提交版本**（或加锁等待），并可能触发写冲突检测，这个过程不受事务开始时生成的 ReadView 限制。
3.  **幻读（Phantom Read）的定义与解决：** RR 隔离级别通过 **Next-Key Lock（临键锁）** 机制来防止幻读，这属于“锁定读”范畴，与 ReadView 提供的“无锁快照读”是互补的两种机制。
4.  **长时间事务的影响：** 由于 RR 事务需要维护其开始时的 ReadView，可能导致 Undo Log 无法被及时清理，从而占用过多存储空间。监控长事务至关重要。

---

## **6. 总结**

在 InnoDB 的 RR 隔离级别中，**“事务开始时生成 ReadView”** 是 MVCC 实现可重复读语义的基石。它通过为事务在首次查询时固化一个一致性快照，并在此后所有的普通读操作中复用此快照来判定数据可见性，从而确保了在同一事务内查询结果的可重复性。理解这一机制，对于正确设计高并发应用、分析事务现象和进行数据库调优都具有重要意义。
