# Spring三级缓存解决循环依赖技术文档

## 1. 概述

### 1.1 什么是循环依赖
循环依赖指的是两个或多个Bean之间相互依赖，形成闭环的依赖关系。例如：
- Bean A 依赖于 Bean B
- Bean B 依赖于 Bean A

### 1.2 循环依赖的类型
1. **构造函数循环依赖** - 无法解决，Spring会抛出BeanCurrentlyInCreationException
2. **Setter方法/字段循环依赖** - 可以通过三级缓存解决
3. **原型模式循环依赖** - 无法解决，与单例模式处理机制不同

## 2. Spring容器创建Bean的流程

### 2.1 主要步骤
```
1. 实例化 (Instantiate)
2. 属性填充 (Populate)
3. 初始化 (Initialize)
4. 销毁 (Destroy)
```

### 2.2 循环依赖产生场景
```java
@Service
public class ServiceA {
    @Autowired
    private ServiceB serviceB;
}

@Service
public class ServiceB {
    @Autowired
    private ServiceA serviceA;
}
```

## 3. 三级缓存机制详解

### 3.1 三级缓存定义
Spring使用三个Map容器来管理Bean的不同状态：

```java
// DefaultSingletonBeanRegistry.java 中的定义
// 一级缓存：存放完整的单例Bean
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

// 二级缓存：存放早期暴露的Bean（未完成属性填充）
private final Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>(16);

// 三级缓存：存放Bean工厂，用于创建早期Bean引用
private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);
```

### 3.2 各级缓存的作用

#### 一级缓存 (singletonObjects)
- **作用**：存放完全初始化完成的单例Bean
- **时机**：Bean完成所有初始化步骤后放入
- **特点**：可以直接使用的完整Bean

#### 二级缓存 (earlySingletonObjects)
- **作用**：存放早期暴露的Bean对象
- **时机**：Bean实例化后，但未完成属性填充时
- **特点**：避免重复创建代理对象

#### 三级缓存 (singletonFactories)
- **作用**：存放创建Bean的工厂对象
- **时机**：Bean实例化后立即放入
- **特点**：支持AOP代理对象的创建

## 4. 解决循环依赖的核心流程

### 4.1 时序流程（以ServiceA和ServiceB为例）

```
1. 创建ServiceA
   │
2. 实例化ServiceA（通过构造函数创建原始对象）
   │
3. 将ServiceA的ObjectFactory放入三级缓存
   │
4. 填充ServiceA的属性（发现需要ServiceB）
   │
5. 创建ServiceB
   │    │
   │    6. 实例化ServiceB
   │    │
   │    7. 将ServiceB的ObjectFactory放入三级缓存
   │    │
   │    8. 填充ServiceB的属性（发现需要ServiceA）
   │    │
   │    9. 从三级缓存获取ServiceA的ObjectFactory
   │    │
   │    10. 通过ObjectFactory获取ServiceA的早期引用
   │    │      （如果需要AOP，会创建代理对象）
   │    │
   │    11. 将ServiceA早期引用放入二级缓存，清除三级缓存
   │    │
   │    12. ServiceB完成属性填充和初始化
   │    │
   │    13. ServiceB放入一级缓存
   │
14. ServiceA继续属性填充（获得完整的ServiceB）
   │
15. ServiceA完成初始化
   │
16. ServiceA从二级缓存升级到一级缓存
```

### 4.2 关键代码逻辑

```java
// AbstractAutowireCapableBeanFactory.doCreateBean()
protected Object doCreateBean(String beanName, RootBeanDefinition mbd, Object[] args) {
    // 1. 实例化Bean
    BeanWrapper instanceWrapper = createBeanInstance(beanName, mbd, args);
    final Object bean = instanceWrapper.getWrappedInstance();
    
    // 2. 判断是否支持提前暴露
    boolean earlySingletonExposure = (mbd.isSingleton() && 
                                     this.allowCircularReferences && 
                                     isSingletonCurrentlyInCreation(beanName));
    
    if (earlySingletonExposure) {
        // 3. 添加ObjectFactory到三级缓存
        addSingletonFactory(beanName, () -> getEarlyBeanReference(beanName, mbd, bean));
    }
    
    // 4. 属性填充（可能触发依赖Bean的创建）
    populateBean(beanName, mbd, instanceWrapper);
    
    // 5. 初始化
    exposedObject = initializeBean(beanName, exposedObject, mbd);
    
    return exposedObject;
}

// DefaultSingletonBeanRegistry.getSingleton()
public Object getSingleton(String beanName) {
    // 1. 从一级缓存获取
    Object singletonObject = this.singletonObjects.get(beanName);
    if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
        synchronized (this.singletonObjects) {
            // 2. 从二级缓存获取
            singletonObject = this.earlySingletonObjects.get(beanName);
            if (singletonObject == null) {
                // 3. 从三级缓存获取ObjectFactory
                ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);
                if (singletonFactory != null) {
                    // 4. 通过ObjectFactory创建Bean
                    singletonObject = singletonFactory.getObject();
                    // 5. 移动到二级缓存
                    this.earlySingletonObjects.put(beanName, singletonObject);
                    this.singletonFactories.remove(beanName);
                }
            }
        }
    }
    return singletonObject;
}
```

## 5. AOP代理与循环依赖

### 5.1 代理对象的处理
当Bean需要AOP代理时，三级缓存中的ObjectFactory会创建代理对象：

```java
// AbstractAutowireCapableBeanFactory.getEarlyBeanReference()
protected Object getEarlyBeanReference(String beanName, RootBeanDefinition mbd, Object bean) {
    Object exposedObject = bean;
    if (!mbd.isSynthetic() && hasInstantiationAwareBeanPostProcessors()) {
        for (BeanPostProcessor bp : getBeanPostProcessors()) {
            if (bp instanceof SmartInstantiationAwareBeanPostProcessor) {
                SmartInstantiationAwareBeanPostProcessor ibp = 
                    (SmartInstantiationAwareBeanPostProcessor) bp;
                exposedObject = ibp.getEarlyBeanReference(exposedObject, beanName);
            }
        }
    }
    return exposedObject;
}
```

### 5.2 为什么需要三级缓存而不是两级
1. **分离关注点**：ObjectFactory专注于创建对象（可能是原始对象或代理对象）
2. **避免重复创建代理**：确保同一个Bean的代理只创建一次
3. **支持多种后处理器**：不同的SmartInstantiationAwareBeanPostProcessor可能返回不同的代理

## 6. 特殊情况处理

### 6.1 构造函数循环依赖（无法解决）
```java
@Component
public class ConstructorA {
    private ConstructorB b;
    
    @Autowired
    public ConstructorA(ConstructorB b) {
        this.b = b;
    }
}

@Component
public class ConstructorB {
    private ConstructorA a;
    
    @Autowired
    public ConstructorB(ConstructorA a) {
        this.a = a;
    }
}
```
- **原因**：Bean实例化之前无法提前暴露引用

### 6.2 原型模式的循环依赖（无法解决）
- **原因**：原型Bean不缓存，每次都会重新创建

### 6.3 使用@Async的循环依赖
```java
@Service
public class AsyncServiceA {
    @Autowired
    private AsyncServiceB serviceB;
    
    @Async
    public void asyncMethod() {
        // 异步方法
    }
}
```
- **问题**：@Async通过后处理器创建代理，可能破坏三级缓存机制
- **解决方案**：避免在循环依赖中使用@Async

## 7. 配置与优化

### 7.1 关闭循环依赖支持
```properties
# application.properties
spring.main.allow-circular-references=false
```

### 7.2 最佳实践
1. **代码设计层面避免循环依赖**
2. 使用setter注入代替字段注入
3. 提取公共逻辑到第三方Bean
4. 使用@Lazy延迟加载
5. 使用ApplicationContext.getBean()手动获取（不推荐）

### 7.3 @Lazy注解的使用
```java
@Service
public class ServiceA {
    @Autowired
    @Lazy  // 延迟注入，打破循环
    private ServiceB serviceB;
}
```

## 8. 调试与问题排查

### 8.1 常见异常
1. **BeanCurrentlyInCreationException**
   - 可能原因：构造函数循环依赖
   - 解决方案：重构代码，使用setter注入

2. **NullPointerException**
   - 可能原因：AOP代理与循环依赖冲突
   - 解决方案：检查@Async、@Transactional等注解

## 9. 总结

### 9.1 三级缓存机制要点
| 缓存级别 | 存储内容 | 生命周期 | 作用 |
|---------|---------|---------|------|
| 一级缓存 | 完整Bean | Bean初始化完成后 | 提供完整的单例Bean |
| 二级缓存 | 早期Bean | Bean属性填充阶段 | 避免重复创建代理 |
| 三级缓存 | Bean工厂 | Bean实例化后立即 | 支持AOP代理创建 |

### 9.2 设计思想
1. **空间换时间**：通过三级缓存减少重复计算
2. **关注点分离**：不同缓存处理不同阶段的Bean
3. **延迟创建**：ObjectFactory支持按需创建代理
4. **状态升级**：Bean从三级→二级→一级缓存逐步完善

### 9.3 局限性
1. 只能解决单例模式的setter/字段循环依赖
2. 增加系统复杂性
3. 可能掩盖设计缺陷（循环依赖通常意味着设计问题）
