# AQS独占锁与共享锁实现差异

## 1. 概述

AbstractQueuedSynchronizer（AQS）是Java并发包的核心同步器框架，提供了两种主要的同步模式：**独占锁**（Exclusive）和**共享锁**（Shared）。这两种模式在实现原理、应用场景和使用方式上存在显著差异。

## 2. 核心差异对比

| 特性 | 独占锁（Exclusive） | 共享锁（Shared） |
|------|-------------------|-----------------|
| **持有者数量** | 同一时刻只能被一个线程持有 | 可被多个线程同时持有 |
| **典型实现** | ReentrantLock、WriteLock | Semaphore、CountDownLatch、ReadLock |
| **应用场景** | 写操作、互斥访问 | 读操作、资源共享 |
| **AQS方法** | tryAcquire/tryRelease | tryAcquireShared/tryReleaseShared |
| **传播模式** | 不传播 | 可传播唤醒后续共享节点 |

## 3. 实现机制差异

### 3.1 状态（State）管理

**独占锁状态语义：**
- 0：锁未被持有
- 1：锁被一个线程持有
- >1：重入次数（如ReentrantLock）

```java
// 独占锁典型状态转换
protected boolean tryAcquire(int arg) {
    // 使用CAS操作将state从0改为1
    // 记录当前持有线程，支持重入
}
```

**共享锁状态语义：**
- 资源数量或剩余许可数
- 如Semaphore中state表示可用许可数量
- CountDownLatch中state表示剩余计数

```java
// 共享锁典型状态转换
protected int tryAcquireShared(int arg) {
    // 计算剩余资源：state - arg
    // 返回正数表示获取成功，负数表示需要入队
}
```

### 3.2 等待队列节点差异

#### 独占锁节点：
```java
// Node.EXCLUSIVE标记
static final class Node {
    static final Node EXCLUSIVE = null; // 独占模式
    // 唤醒时只唤醒下一个节点（通常）
}
```

#### 共享锁节点：
```java
// Node.SHARED标记
static final class Node {
    static final Node SHARED = new Node(); // 共享模式
    // 唤醒时会传播唤醒后续共享节点
}
```

### 3.3 唤醒机制差异

**独占锁唤醒：**
```java
// 释放锁时只唤醒下一个节点
private void unparkSuccessor(Node node) {
    // 找到等待队列中的下一个节点
    // 唤醒该节点对应的线程
}
```

**共享锁唤醒：**
```java
// 释放共享资源时会传播唤醒
private void doReleaseShared() {
    // 唤醒后继节点
    // 后继节点被唤醒后，可能继续传播唤醒
    // 实现"唤醒风暴"效应
}
```

## 4. 核心方法实现对比

### 4.1 获取锁流程

**独占锁获取（acquire）：**
```java
public final void acquire(int arg) {
    if (!tryAcquire(arg) &&
        acquireQueued(addWaiter(Node.EXCLUSIVE), arg))
        selfInterrupt();
}
```

**共享锁获取（acquireShared）：**
```java
public final void acquireShared(int arg) {
    if (tryAcquireShared(arg) < 0)
        doAcquireShared(arg);
}
```

### 4.2 释放锁流程

**独占锁释放：**
```java
public final boolean release(int arg) {
    if (tryRelease(arg)) {
        Node h = head;
        if (h != null && h.waitStatus != 0)
            unparkSuccessor(h);
        return true;
    }
    return false;
}
```

**共享锁释放：**
```java
public final boolean releaseShared(int arg) {
    if (tryReleaseShared(arg)) {
        doReleaseShared(); // 传播唤醒
        return true;
    }
    return false;
}
```

## 5. 具体实现示例

### 5.1 独占锁实现模板
```java
class ExclusiveLock extends AbstractQueuedSynchronizer {
    protected boolean tryAcquire(int acquires) {
        Thread current = Thread.currentThread();
        int c = getState();
        if (c == 0) {
            if (compareAndSetState(0, acquires)) {
                setExclusiveOwnerThread(current);
                return true;
            }
        } else if (current == getExclusiveOwnerThread()) {
            int nextc = c + acquires; // 重入
            setState(nextc);
            return true;
        }
        return false;
    }
    
    protected boolean tryRelease(int releases) {
        int c = getState() - releases;
        if (Thread.currentThread() != getExclusiveOwnerThread())
            throw new IllegalMonitorStateException();
        boolean free = false;
        if (c == 0) {
            free = true;
            setExclusiveOwnerThread(null);
        }
        setState(c);
        return free;
    }
}
```

### 5.2 共享锁实现模板
```java
class SharedLock extends AbstractQueuedSynchronizer {
    protected int tryAcquireShared(int acquires) {
        for (;;) {
            int available = getState();
            int remaining = available - acquires;
            if (remaining < 0 || 
                compareAndSetState(available, remaining))
                return remaining; // 负数表示失败
        }
    }
    
    protected boolean tryReleaseShared(int releases) {
        for (;;) {
            int current = getState();
            int next = current + releases;
            if (next < current) // 溢出检查
                throw new Error("Maximum permit count exceeded");
            if (compareAndSetState(current, next))
                return true;
        }
    }
}
```

## 6. 混合模式：读写锁的实现

ReentrantReadWriteLock是两种模式的结合：
- 读锁：共享模式
- 写锁：独占模式

**状态分割：**
```java
// 高16位：共享锁计数（读锁）
// 低16位：独占锁计数（写锁）
static final int SHARED_SHIFT   = 16;
static final int SHARED_UNIT    = (1 << SHARED_SHIFT);
static final int MAX_COUNT      = (1 << SHARED_SHIFT) - 1;
static final int EXCLUSIVE_MASK = (1 << SHARED_SHIFT) - 1;
```

## 7. 性能与适用场景

### 独占锁适用场景：
- 写多读少的场景
- 需要互斥访问的资源
- 需要严格的一致性保证

### 共享锁适用场景：
- 读多写少的场景
- 资源池管理（连接池、线程池）
- 并发计数器

## 8. 总结

| 维度 | 独占锁 | 共享锁 |
|------|--------|--------|
| **并发度** | 低（串行化） | 高（并行化） |
| **实现复杂度** | 相对简单 | 需要处理传播唤醒 |
| **公平性保证** | 容易实现 | 需要考虑资源分配策略 |
| **典型使用** | 互斥操作 | 资源池、信号量 |

理解AQS中独占锁和共享锁的实现差异，有助于：
1. 选择合适的同步机制
2. 实现自定义的同步器
3. 优化并发程序性能
4. 诊断死锁和性能问题

两种模式在AQS框架中通过统一的等待队列管理，共享大部分基础设施，但通过不同的方法模板和节点类型实现了不同的同步语义，体现了AQS框架设计的巧妙性和灵活性。
