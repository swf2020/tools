# MySQL自适应哈希索引（AHI）技术文档

## 1. 概述

### 1.1 什么是自适应哈希索引
自适应哈希索引（Adaptive Hash Index，AHI）是InnoDB存储引擎的一种内部优化机制，用于自动创建和管理基于内存的哈希索引，以加速B+树索引的查找操作。

### 1.2 设计目的
- **减少B+树遍历开销**：将频繁访问的索引页地址转换为哈希表快速查找
- **自动适应工作负载**：根据访问模式动态创建和删除哈希条目
- **完全透明**：无需用户干预，由InnoDB自动管理

## 2. 工作原理

### 2.1 触发条件
AHI在以下条件满足时自动创建：
```sql
-- 当某个索引页被连续访问次数达到阈值时
-- InnoDB_monitor输出示例：
INSERT BUFFER AND ADAPTIVE HASH INDEX
-------------------------------------
Ibuf: size 1, free list len 0, seg size 2, 0 merges
Hash table size 276671, node heap has 1 buffer(s)
Hash table size 276671, node heap has 0 buffer(s)
Hash table size 276671, node heap has 0 buffer(s)
```

### 2.2 哈希键构成
```
哈希键 = (数据库名 + 表名 + 索引列值)的哈希值
哈希值 → 索引页的物理地址（直接指向数据页）
```

### 2.3 内存结构
```c
// 伪代码表示AHI内存结构
struct adaptive_hash_index {
    hash_table_t *table;          // 哈希表
    rw_lock_t *latch;             // 读写锁
    ulint n_cells;                // 哈希槽数量
    ulint n_ref_count;            // 引用计数
};
```

## 3. 配置参数

### 3.1 核心参数
```ini
# my.cnf配置文件示例
[mysqld]
# 启用/禁用AHI（默认ON）
innodb_adaptive_hash_index = ON

# AHI分区数（默认8，减少锁争用）
innodb_adaptive_hash_index_parts = 8

# 监控相关
innodb_monitor_enable = module_adaptive_hash
```

### 3.2 参数说明
| 参数名 | 默认值 | 说明 |
|--------|--------|------|
| `innodb_adaptive_hash_index` | ON | 全局开关 |
| `innodb_adaptive_hash_index_parts` | 8 | 哈希分区数，减少锁争用 |
| `innodb_adaptive_hash_index_parts` | 8 | 取值范围1-512 |

## 4. 适用场景

### 4.1 有效场景
- ✅ **等值查询频繁**
  ```sql
  SELECT * FROM users WHERE user_id = 123;
  ```
- ✅ **索引覆盖查询**
  ```sql
  SELECT id FROM orders WHERE status = 'SHIPPED';
  ```
- ✅ **高并发点查询**
- ✅ **内存充足的环境**

### 4.2 不适用场景
- ❌ **范围查询为主**
  ```sql
  SELECT * FROM logs WHERE create_time BETWEEN '2024-01-01' AND '2024-01-31';
  ```
- ❌ **写密集型负载**
- ❌ **内存严重不足**
- ❌ **索引列值非常长**

## 5. 监控与诊断

### 5.1 状态查看
```sql
-- 查看AHI使用情况
SHOW ENGINE INNODB STATUS\G
-- 在输出中查找"INSERT BUFFER AND ADAPTIVE HASH INDEX"部分

-- 通过系统表监控
SELECT * FROM information_schema.INNODB_METRICS
WHERE NAME LIKE '%adaptive_hash%';

-- 性能模式监控
SELECT * FROM performance_schema.events_waits_summary_global_by_event_name
WHERE EVENT_NAME LIKE '%adaptive_hash%';
```

### 5.2 关键指标
```sql
-- AHI命中率计算
SELECT 
    (1 - (Variable_value / 
        (SELECT Variable_value 
         FROM information_schema.GLOBAL_STATUS 
         WHERE Variable_name = 'innodb_buffer_pool_read_requests')
    )) * 100 AS 'AHI Hit Rate (%)'
FROM information_schema.GLOBAL_STATUS
WHERE Variable_name = 'innodb_buffer_pool_reads';
```

## 6. 性能优化建议

### 6.1 启用条件
```sql
-- 判断是否应启用AHI
SELECT 
    -- 如果点查询比例高，考虑启用
    CASE WHEN point_select_ratio > 0.7 THEN 'ENABLE AHI'
         ELSE 'EVALUATE FURTHER' 
    END AS recommendation
FROM (
    SELECT 
        SUM(Variable_value) point_selects,
        (SELECT SUM(Variable_value) 
         FROM information_schema.GLOBAL_STATUS 
         WHERE Variable_name IN ('Com_select', 'Com_insert', 'Com_update', 'Com_delete')) total_ops,
        SUM(Variable_value) / 
        (SELECT SUM(Variable_value) 
         FROM information_schema.GLOBAL_STATUS 
         WHERE Variable_name IN ('Com_select', 'Com_insert', 'Com_update', 'Com_delete')) point_select_ratio
    FROM information_schema.GLOBAL_STATUS
    WHERE Variable_name = 'Handler_read_key'
) t;
```

### 6.2 调整策略
1. **增加分区数**（减少锁争用）：
   ```sql
   SET GLOBAL innodb_adaptive_hash_index_parts = 16;
   ```

2. **监控内存使用**：
   ```sql
   -- 检查AHI内存使用
   SELECT * FROM sys.memory_global_by_current_bytes
   WHERE event_name LIKE '%adaptive_hash%';
   ```

3. **禁用场景**：
   ```ini
   # 在my.cnf中禁用AHI
   innodb_adaptive_hash_index = OFF
   ```

## 7. 故障排查

### 7.1 常见问题
1. **CPU使用率高**
   - 可能原因：AHI锁争用
   - 解决方案：增加`innodb_adaptive_hash_index_parts`

2. **内存增长**
   - 可能原因：AHI占用过多内存
   - 解决方案：监控AHI大小，必要时禁用

3. **性能下降**
   ```sql
   -- 检查AHI效率
   SELECT 
       (SELECT Variable_value 
        FROM information_schema.GLOBAL_STATUS 
        WHERE Variable_name = 'innodb_adaptive_hash_searches') as hash_searches,
       (SELECT Variable_value 
        FROM information_schema.GLOBAL_STATUS 
        WHERE Variable_name = 'innodb_adaptive_hash_searches_btree') as btree_searches,
       (SELECT Variable_value 
        FROM information_schema.GLOBAL_STATUS 
        WHERE Variable_name = 'innodb_adaptive_hash_pages_added') as pages_added,
       (SELECT Variable_value 
        FROM information_schema.GLOBAL_STATUS 
        WHERE Variable_name = 'innodb_adaptive_hash_pages_removed') as pages_removed;
   ```

### 7.2 诊断步骤
```bash
# 1. 检查当前AHI状态
mysqladmin ext | grep -i adaptive_hash

# 2. 监控锁等待
pt-deadlock-logger --user=root --password=*** h=localhost

# 3. 分析工作负载
pt-query-digest slow.log
```

| MySQL版本 | AHI改进 |
|-----------|---------|

## 9. 最佳实践总结

1. **监控优先**：始终监控AHI命中率和锁争用
2. **适度调整**：根据负载特征调整分区数
3. **场景匹配**：仅在点查询为主的场景启用
4. **版本升级**：新版本通常有更好的AHI实现
5. **备选方案**：考虑使用覆盖索引作为AHI的替代
